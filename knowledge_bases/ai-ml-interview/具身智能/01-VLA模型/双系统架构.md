# 双系统架构（System 2 + System 1）

> 本文详解 2025-2026 年 VLA 的主流范式：**大脑-小脑双系统双频架构**。包括设计原理、Figure Helix 详解、Chunk Prediction / Action Chunking / 时序集成、VLM 冻结 vs 半冻结 vs 联合训练的取舍。

## 一、为什么要双系统

### 1.1 单系统的瓶颈

2023-2024 年的 VLA（RT-2、OpenVLA、π₀ 单塔版本）存在一个结构性矛盾：

- **VLM 主干大**（3-7B+）：理解语言和场景需要通用常识，必然参数多
- **控制要求高频**（>100 Hz）：接触任务、动态平衡、反射性避让需要快速闭环
- 两个要求在同一前向路径上**不可兼得**

结果是 π₀ 单塔 10-50 Hz、Gemini Robotics 10-30 Hz，都远未达到动态任务所需的 100-200 Hz。

### 1.2 Kahneman 的"快与慢"启发

卡尼曼《思考，快与慢》中人类的**System 1（快，直觉）+ System 2（慢，推理）** 直接启发了机器人架构：

```
人类:      System 1 (直觉, <100ms)     +    System 2 (推理, 秒级)
机器人:    动作专家 (100-200 Hz)       +    VLM 主干 (7-9 Hz)
```

### 2.1 异步双频关键

- **S2 与 S1 异步运行**：S2 算 1 次 → S1 用这份 latent 跑 ~20 次（9 Hz vs 200 Hz）
- **S2 latent 有缓存**：S1 不等 S2，有旧 latent 就用旧的
- **pipelining**：S2 在算 t+1 时刻的 latent，S1 在用 t 时刻的 latent 出动作

## 三、Figure Helix 详解（2025.02）

Helix 是双系统双频架构的工程标杆，也是首个公开部署的此类 VLA。

### 3.1 架构细节

```
System 2 (S2):
  - 主干: 7B VLM (internet-scale 预训练)
  - 频率: 9 Hz
  - 输入: 多相机 RGB (3-4 路) + 语言指令
  - 输出: 连续 latent vector (256 维)
  - 参数冻结状态: 半冻结（视觉塔冻结，上层 Transformer 微调）

System 1 (S1):
  - 主干: 80M Transformer
  - 频率: 200 Hz
  - 输入:
      * S2 latent (256-d)
      * 本体状态 (关节角、速度、末端力)
      * 历史动作 (短窗口)
  - 输出: 连续动作 chunk (全身控制, 35-DoF 上半身)
  - 动作表示: 连续回归（非扩散）

训练:
  - ~500 小时高质量 Figure 02 遥操作
  - 单一策略覆盖: 抓取/放置/递送/开柜门/家居
  - 无需任务特定微调
  - S2 与 S1 联合训练（S2 视觉塔冻结）
```

### 3.2 为什么 S1 选 Transformer 而非 FM

Figure 选择**回归 Transformer** 而非 Flow Matching，原因：

1. **200 Hz 闭环要求极致低延迟**：FM 即便 5 步 Euler 也需要 5 次前向
2. **S2 latent 已承担条件编码**：S1 不需要做"从复杂分布采样"，只需"latent + 状态 → 动作"
3. **多模态分布由 S2 承担**：S2 在子目标层面给出不同选择，S1 只跟随 S2 给的 latent 执行

对比 π₀.5 / GR00T N1 选择 FM/Diffusion，是因为它们的 S1 要承担更多分布建模职责。

### 3.3 Helix 工程亮点

1. **单一神经网络全身协调**：双臂 + 躯干 + 手 + 头，一个模型全部输出
2. **双机器人协作**：两台 Figure 02 协同（一递一放），latent 空间天然支持"跨本体"
3. **端到端双频训练**：S1/S2 联合训练，不是两阶段流水线
4. **商业闭环**：BMW 工厂部署持续产生真机数据

### 3.4 Helix v2（2025 H2）

部署 Figure 03，改进方向：

- 更长 horizon 推理（S2 加入短期记忆）
- 家庭场景（洗碗、叠衣）
- 可能引入 FM 动作头做 chunk 精修

## 四、其他双系统 VLA

### 4.1 NVIDIA GR00T N1（2025.03）

```
S2: Eagle-2 VLM (2B), 7 Hz
S1: Diffusion Transformer (DiT, 1B), 30+ Hz
    动作 chunk 16 步
```

特点：S1 用 DiT，适合人形多模态动作分布。

### 4.2 Physical Intelligence π₀.5（2025.04）

虽然 π₀ 原版是单塔设计，π₀.5 实际上做了快慢解耦：

```
高层: VLM 自主语言规划（discrete language action）
低层: Flow Matching 动作专家（50 步 chunk）
```

介于单塔和典型双系统之间。

### 4.3 Gemini Robotics（2025.03）

```
S2: Gemini 2 Vision
S1: Diffusion Transformer
频率: 10-30 Hz
```

频率较低是因为是桌面双臂场景，不需要 200 Hz。

## 五、Chunk Prediction / Action Chunking

### 5.1 什么是 Action Chunking

**一次前向预测未来 N 步动作**（N=16、50 是常见选择）。每步不重新前向，而是执行 chunk 中的下一步。

```
t=0   前向 → chunk_0 = [a_0, a_1, ..., a_{49}]  (50 步)
t=0   执行 a_0
t=1   执行 a_1
...
t=?   再次前向 → chunk_? = [...]
```

### 5.2 为什么有效

1. **降低推理频次**：50 步 chunk，10 Hz 前向即可达到 500 Hz 实际执行
2. **连贯性**：chunk 内动作是模型一次"想明白"的，不会互相抖动
3. **长 horizon 覆盖**：1-2 秒的未来窗口足够覆盖一次"抓取-举起-放置"子步

### 5.3 何时重新前向

三种策略：

| 策略 | 说明 | 代表 |
|------|------|------|
| **Open loop** | 执行完整 chunk 再前向 | 早期 ALOHA |
| **固定频率** | 每 K 步强制前向（K=1 即每步） | π₀ (50 步 chunk, 10 Hz 前向 = 500 虚拟 Hz) |
| **Temporal Ensembling** | 每步前向，对重叠 chunk 做加权 | ALOHA、Diffusion Policy |

### 5.4 Temporal Ensembling 详解

每一步都前向一次得到新 chunk，不同时间步的 chunk 对同一动作时刻 `t` 有多个预测，做指数加权：

```
给定时刻 t 被 k=1..K 个 chunk 预测出 {a_t^{(1)}, ..., a_t^{(K)}}
加权:  a_t = Σ_k w_k · a_t^{(k)},    w_k ∝ exp(-k · m)
   (最新的预测权重最高)
```

**优点**：平滑接缝、减少抖动。
**代价**：每步都前向一次，成本高（除非 S2/S1 解耦，让 S1 高频前向）。

## 六、VLM 冻结 vs 半冻结 vs 联合训练

双系统架构下的一个核心设计抉择：S2 的 VLM 参数是否微调？

### 6.1 三种策略对比

| 策略 | 说明 | 优 | 劣 | 代表 |
|------|------|----|----|------|
| **完全冻结** | VLM 参数一点不动，只训 S1 | 保留常识，训练快 | 视觉/语言表示不适配机器人场景 | 某些早期实验 |
| **半冻结** | 视觉塔冻结，上层 Transformer 微调 | 平衡常识与适配 | 实现复杂 | **Figure Helix**, π₀.5 |
| **联合训练** | S2 和 S1 全参数联合训练 | 表示最贴合任务 | 可能破坏 VLM 常识（灾难性遗忘） | π₀ 原版、OpenVLA |

### 6.2 实践建议

- **数据规模小（<1000 小时）**：半冻结或冻结，避免过拟合
- **数据规模大（>10000 小时）**：可以联合训练，但要加 VLM 正则化（KL 到原 VLM）
- **多任务 / 跨本体**：联合训练 + 共享 S2，任务特化的是 S1

### 6.3 灾难性遗忘的应对

- **Replay buffer**：训练时混入一小部分 VLM 预训练数据
- **LoRA 微调**：只学低秩增量，不动主干
- **KL 正则**：保留 VLM 输出分布
- **两阶段训练**：先 S1 单独训，再联合微调

## 七、双系统的工程细节

### 7.1 Latent 传递方式

S2 → S1 的接口可以是：

1. **CLS token / Pooling 向量**：固定维度（256-2048），最常见
2. **Token 序列**：S1 用 cross-attention 消费 S2 的所有输出 token（更丰富但更贵）
3. **Language tokens**：S2 输出自然语言子目标，S1 用语言 encoder 再编码（π₀.5 的 discrete language action 路线）

### 7.2 异步调度与缓存

```
时间线（毫秒级）:

 0ms   S2 前向开始，S1 用上一 latent 跑
100ms  S2 前向仍在进行，S1 已跑 20 步
110ms  S2 前向完成，新 latent 就位
111ms  S1 换用新 latent 继续跑

关键: S1 从不阻塞等 S2
```

实际部署中常用**两条异步 CUDA stream**或**S2 跑 GPU 0、S1 跑 GPU 1** 的方式。

### 7.3 本体状态的注入

S1 的输入除了 S2 latent，还有：

- 关节角、关节速度
- 末端位姿、末端速度
- 末端力/力矩（如有）
- IMU（移动底盘）
- 历史动作（短窗）

本体状态高频变化，是 S1 高频反应的基础。

### 7.4 安全与兜底

双系统下，安全层通常放在 S1 输出和低级控制器之间：

- 碰撞检测（joint limit、self-collision、外部障碍）
- 力/扭矩饱和监控
- 紧急停机（Watch-dog）
- 关节空间平滑（低通 / 限加加速度）

## 八、双系统架构的局限

1. **S2 latent 信息量不足时 S1 无能为力**：如果 S2 低频（9 Hz）遗漏了一个快速移动的物体，S1 也感知不到
2. **异步工程复杂度**：流水线、缓存、时序同步需要精心实现
3. **训练对齐困难**：S1 高频样本 vs S2 低频样本，如何配对？
4. **评估麻烦**：单独评估 S2 或 S1 意义不大，必须端到端评估

这些是 2026 年研究的活跃方向。

## 九、未来：三系统 / 多系统？

一些前沿提出**三系统/多系统** 设想：

```
System 3: 世界模型 rollout / 反事实规划 (分钟级)
System 2: VLM 推理 (秒级)
System 1: 动作专家 (10ms 级)
System 0: 反射层 (1ms 级安全兜底)
```

代表思路：
- **V-JEPA 2 + 下层 VLA**（规划-执行分离）
- **π₀.5 + Cosmos** 的世界模型耦合
- 人形机器人的多层控制（全身平衡 + 操控 + 表达）
