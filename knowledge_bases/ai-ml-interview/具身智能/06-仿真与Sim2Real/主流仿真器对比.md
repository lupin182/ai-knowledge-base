## 一、NVIDIA Isaac 生态

### Isaac Sim

- 基于 Omniverse 平台，物理引擎 PhysX 5，支持刚体、柔体、流体、布料
- **RTX 光线追踪**，PBR 材质，视觉接近照片级
- 传感器建模完整：RGB-D、LiDAR、IMU、力/力矩
- 域随机化组件化：材质、光照、相机、物理参数一键开关

### Isaac Lab（原 Orbit，2023 起整合）

Isaac Sim 之上的 RL/IL 训练框架：

```python
import isaaclab

env = isaaclab.make("Isaac-Franka-Reach-v0", num_envs=4096)  # 单卡 4096 并行
obs = env.reset()
for step in range(10000):
    actions = policy(obs)
    obs, reward, done, info = env.step(actions)  # GPU 并行 step
```

优势：
- **端到端 GPU**：观测、物理、渲染全部 GPU 内完成，避免 CPU-GPU 搬运
- 丰富任务库：Reach、Pick-and-Place、Locomotion、Dexterous Hand
- 与 Isaac Gym (legacy) 相比支持 Isaac Sim 的高保真渲染

### Isaac GR00T

面向人形机器人的基础模型框架（详见 [../01-VLA模型/](../01-VLA模型/)）：
- GR00T N1 基础模型：VLM 高层规划 + 扩散策略底层控制
- GR00T Dreams (2025)：世界模型合成数据 pipeline

## 二、MuJoCo 与 MJX

### MuJoCo 核心特点

- **Multi-Joint dynamics with Contact**：凸优化求解接触力，物理精度高
- **MJCF XML 模型定义**：声明式
- C 语言实现，单核数千 FPS
- 2021 年 DeepMind 收购后开源，成为学术界默认

```xml
<mujoco>
  <worldbody>
    <body name="robot" pos="0 0 1">
      <joint name="hip" type="hinge" axis="0 1 0"/>
      <geom type="capsule" size="0.05 0.3"/>
      <body name="shin" pos="0 0 -0.6">
        <joint name="knee" type="hinge" axis="0 1 0" range="0 150"/>
        <geom type="capsule" size="0.04 0.3"/>
      </body>
    </body>
  </worldbody>
  <actuator>
    <motor joint="hip" ctrlrange="-1 1"/>
    <motor joint="knee" ctrlrange="-1 1"/>
  </actuator>
</mujoco>
```

### 经典 Benchmark

| 环境 | obs 维度 | action 维度 | 任务 |
|------|---------|------------|------|
| HalfCheetah | 17 | 6 | 跑步 |
| Ant | 111 | 8 | 四足行走 |
| Humanoid | 376 | 17 | 双足行走 |
| Shadow Hand | 211 | 20 | 灵巧操作 |
| Adroit (DAPG) | ~40 | 24-30 | 工具使用 |

### MJX：JAX 上的可微分 MuJoCo

```python
import mujoco
from mujoco import mjx
import jax

model = mujoco.MjModel.from_xml_path("humanoid.xml")
mjx_model = mjx.put_model(model)

@jax.vmap
def step(state, action):
    return mjx.step(mjx_model, state.replace(ctrl=action))

batch_states = jax.vmap(lambda k: mjx.make_data(mjx_model))(keys)  # 数千并行
```

MJX 使 MuJoCo 同时获得 **GPU 并行** + **可微分**，成为 2024-2026 学术界最流行的 RL 研究环境之一。

## 四、SAPIEN 与 ManiSkill 3

### SAPIEN

- 基于 **PartNet-Mobility** 数据集：丰富的铰接物体（抽屉、门、水龙头、微波炉）
- PhysX 5 / Warp 双后端
- 支持光线追踪渲染

### ManiSkill 3（2024）

ManiSkill 不是独立仿真器，而是**基于 SAPIEN 的操控 benchmark + 训练框架**。类比关系：

```
ManiSkill  :  SAPIEN      ≈
Isaac Lab  :  Isaac Sim   ≈
Habitat-Lab : Habitat-Sim
```

提供：
- **标准化任务集**：刚体抓取、软体、铰接、双臂协作
- **统一 API**：兼容 Gymnasium
- **多种观测**：RGB-D、点云、状态向量
- **Warp 后端**：GPU 大规模并行，视觉 RL 首次可行

| 版本 | 特点 |
|------|-----|
| ManiSkill 1 | 基础操控 benchmark |
| ManiSkill 2 | 软体、更多任务 |
| **ManiSkill 3** (2024) | GPU 并行（PhysX 5 / Warp），视觉 RL 大规模训练 |

## 五、Genesis（2024 新贵）

2024 年开源的通用仿真器，号称 **10-80x 加速**：

- **多物理引擎后端**：刚体（PhysX）、MPM（软体）、SPH（流体）、FEM、PBD
- **生成式仿真**：自然语言描述生成场景
- **可微分**：支持自动微分梯度优化
- **Taichi 核**：GPU 并行编译

> 2026 年评估：在特定场景（软体、流体、MPM）确实显著加速；刚体操控场景与 Isaac Lab / ManiSkill 3 差距不大；渲染质量尚不及 Isaac Sim RTX。

## 八、底层原理：从 action 到关节运动

做具身智能研究**一般不需要深入物理引擎内部**，但理解 `env.step(action)` 背后发生了什么对调参和 Sim-to-Real 迁移至关重要。

### action → 力矩 → 物理引擎链路

```
policy 输出 actions (归一化的 [-1, 1])
        │
        ▼
┌─────────────────────────┐
│  Action Scaling          │  actions × scale + offset → 目标值
│  (归一化 → 物理量)       │  例: [-1,1] → [-30°, 30°] 关节角度
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│  Actuator Model          │  模拟真实电机的响应
│  (执行器模型)            │  目标值 → 实际施加的力/力矩
└─────────┬───────────────┘
          ▼
┌─────────────────────────┐
│  Physics Engine          │  力/力矩 + 接触 → 下一时刻状态
│  (PhysX / MuJoCo)        │  牛顿-欧拉方程 + 约束求解
└─────────┬───────────────┘
          ▼
    新的 obs, reward
```

### 三种执行器模型

**1. 理想力矩控制**（MuJoCo 经典 benchmark 常用）

```python
τ_applied = action * torque_limit  # 直接施加力矩，无电机动态
```

**2. PD 控制器**（Isaac Lab 默认，最接近真实机器人）

策略输出**目标关节角度**，PD 控制器计算所需力矩：

$$\tau = K_p (q_{target} - q_{current}) + K_d (\dot{q}_{target} - \dot{q}_{current})$$

- $K_p$：刚度（弹簧系数）
- $K_d$：阻尼（阻力系数）
- 真实机器人底层也是 PD 控制器，所以 **Sim-to-Real 接口天然对齐**

**3. 隐式执行器模型**（高保真）

在 PD 基础上增加：电机带宽限制（低通滤波）、摩擦模型、力矩延迟、齿轮间隙。更接近真实电机响应，用于精密操控与人形。

### 控制频率分层

**关键细节：物理仿真频率远高于策略频率**。

```
策略 50 Hz 输出一个目标角度
    → PD 控制器以 500 Hz 持续跟踪（每个 policy step 跑 10 个物理 step）
        → 物理引擎每步求解接触、碰撞、约束
```

常见配置：
- Locomotion：策略 50 Hz，PD 200-500 Hz，物理 500-1000 Hz
- 操控：策略 10-30 Hz，PD 100-200 Hz，物理 240-500 Hz
- 灵巧手：策略 20-30 Hz，PD 500 Hz，物理 1000 Hz

### 观测反馈（传感器建模）

| 观测量 | 物理引擎来源 | 对应真实传感器 |
|--------|-------------|---------------|
| 关节角度 $q$ | 关节状态 | 编码器 |
| 关节角速度 $\dot{q}$ | 状态微分 | 编码器差分 |
| 接触力 | 约束求解器 | 力/力矩传感器 |
| RGB-D 图像 | 光栅化 / 光追渲染 | 相机 |
| IMU 加速度 | 刚体二阶微分 + 重力投影 | IMU |
| LiDAR 点云 | 光线投射 | LiDAR |

通常会叠加噪声：`obs_noisy = obs_clean + noise`，延迟也通过缓冲队列模拟。
