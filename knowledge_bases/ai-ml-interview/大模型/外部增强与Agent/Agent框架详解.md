# Agent 框架详解

Agent（智能体）= LLM + 工具调用 + 自主规划 + 循环执行。不是一次问答，而是**多步自主完成复杂任务**。

## 一、从 ChatBot 到 Agent

| 阶段 | 能力 | 代表 |
|------|------|------|
| **ChatBot** | 对话问答 | ChatGPT 早期 |
| **ChatBot + RAG** | 查知识再回答 | 企业知识问答 |
| **ChatBot + Tool** | 调用工具获取信息/执行操作 | ChatGPT Plugins |
| **Agent** | 自主规划 → 多步执行 → 反思调整 | Claude Code, Devin, Manus |

关键区别：ChatBot 是**一问一答**，Agent 是**接到任务后自主干活**——它会自己规划步骤、选择工具、执行、观察结果、调整计划，循环直到完成。

## 二、Agent 的核心循环：ReAct

ReAct (Reasoning + Acting) 是最基础的 Agent 范式（Yao et al., 2022）：

```
Thought: 我需要先了解这个项目的结构
Action:  list_files("src/")
Observation: ["main.py", "utils.py", "models/", ...]

Thought: 用户要修的 bug 应该在 models/ 下，让我看看
Action:  read_file("src/models/user.py")
Observation: [文件内容...]

Thought: 找到了问题所在，第 42 行的条件判断有误
Action:  edit_file("src/models/user.py", line=42, ...)
Observation: [文件已修改]

Thought: 修完了，跑一下测试确认
Action:  run_tests("test_user.py")
Observation: All tests passed ✓

Thought: Bug 已修复，测试通过，任务完成。
```

**Thought-Action-Observation 循环**是 Agent 的基本骨架。每一步 LLM 都在做推理（Thought），然后决定下一个动作（Action），观察结果（Observation），再推理下一步。

## 三、Agent 的核心组件

```
┌──────────────────────────────────────────┐
│                  Agent                    │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │  LLM     │  │ 规划器    │  │  记忆   │ │
│  │ (大脑)   │  │(Planner) │  │(Memory)│ │
│  └────┬─────┘  └────┬─────┘  └───┬────┘ │
│       │              │             │      │
│       └──────┬───────┘             │      │
│              │                     │      │
│       ┌──────▼──────┐             │      │
│       │  执行引擎    │◄────────────┘      │
│       │ (Executor)  │                    │
│       └──────┬──────┘                    │
│              │                           │
└──────────────┼───────────────────────────┘
               │
    ┌──────────▼──────────┐
    │      工具集          │
    │ RAG / API / 终端     │
    │ 代码执行 / 浏览器    │
    └─────────────────────┘
```

### 3.1 规划（Planning）

Agent 需要把复杂任务分解为可执行的步骤：

| 策略 | 做法 | 代表 |
|------|------|------|
| **ReAct** | 每步推理一次，边做边想 | 最基础 |
| **Plan-then-Execute** | 先制定完整计划，再逐步执行 | HuggingGPT |
| **动态重规划** | 执行中发现问题，修改计划 | Claude Code |

实践中，**动态重规划**效果最好——因为计划赶不上变化，执行过程中经常发现新信息需要调整。

### 3.2 记忆（Memory）

| 类型 | 说明 | 实现方式 |
|------|------|---------|
| **短期记忆** | 当前任务的上下文 | LLM 的 context window |
| **长期记忆** | 跨任务的经验、用户偏好 | 外部存储（文件/数据库） |
| **工作记忆** | 当前推理链、中间结果 | scratchpad / 内部状态 |

Claude Code 的记忆体系就是一个典型例子：
- 短期记忆 = 当前对话上下文
- 长期记忆 = `CLAUDE.md` 文件 + memory 系统
- 工作记忆 = TodoList、Plan

### 3.3 工具使用（Tool Use）

详见 [工具调用与MCP](工具调用与MCP.md)。Agent 通常拥有比普通 ChatBot 更丰富的工具集：

- 文件读写
- 终端命令执行
- 代码搜索与编辑
- 浏览器操作
- API 调用
- RAG 检索

## 四、Agent 的演进历程

### 4.1 概念验证期（2023）

| 项目 | 特点 | 问题 |
|------|------|------|
| **AutoGPT** | 第一个爆火的 Agent，自主完成任务 | 经常跑偏、死循环、花费大量 token |
| **BabyAGI** | 任务分解 + 执行 + 优先级排序 | 规划不够智能 |
| **MetaGPT** | 多 Agent 协作（产品经理、程序员、测试） | 概念有趣但实用性有限 |

这一代的核心问题：**模型能力不够**。GPT-3.5/4 的工具调用准确率和长程规划能力不足以支撑复杂任务。

### 4.2 产品化突破期（2024-2025）

| 项目 | 特点 | 突破 |
|------|------|------|
| **Cursor** | AI 代码编辑器，Agent 辅助编程 | IDE 级集成，开发者体验好 |
| **Devin** | 全自主软件工程 Agent | 首个通过真实工程任务的 Agent |
| **Claude Computer Use** | 直接操控电脑桌面 | 突破了 API 工具的限制 |
| **MCP 发布** | 标准化工具连接协议 | 解决了工具生态碎片化 |

### 4.3 大规模落地期（2025-2026）

| 项目 | 特点 | 意义 |
|------|------|------|
| **Claude Code** | 命令行 Coding Agent，自主读写代码+执行命令 | Anthropic 官方产品，稳定可靠 |
| **OpenHands** (原 OpenDevin) | 开源软件工程 Agent | 开源社区的 Devin 替代品 |
| **Manus** | 通用任务 Agent，强调端到端完成任务 | 2025 爆火，任务完成度高 |
| **Windsurf** | AI 代码编辑器 | Cursor 竞品 |
| **Claude Agent SDK** | Agent 开发框架 | 让开发者构建自定义 Agent |

## 五、为什么 2025-2026 Agent 突然爆发？

2023 年的 AutoGPT 是玩具，2025 年的 Claude Code 是生产工具。差距在哪里？

### 5.1 模型能力跨过临界点

| 能力 | 2023 (GPT-4) | 2025-2026 (Claude 4/GPT-4o+) |
|------|-------------|-------------------------------|
| Tool Calling 准确率 | ~70% | ~95%+ |
| 长程规划 | 3-5 步后失控 | 几十步仍保持连贯 |
| 代码理解 | 单文件级别 | 项目级别（万行代码） |
| 上下文窗口 | 8K-32K | 128K-1M |
| 自我纠错 | 很少能自己发现错误 | 能观察错误并调整 |

**最关键的变化**：模型不再"一步错步步错"，而是能够**在执行中观察、反思、调整**——这是 Agent 从玩具变成工具的分水岭。

### 5.2 MCP 标准化了工具生态

之前：每接入一个工具要写适配代码，生态碎片化。
之后：MCP 统一协议，社区 MCP Server 爆发，Agent 能用的工具急剧增多。

### 5.3 Computer Use 突破了 API 边界

不是所有软件都有 API。Computer Use 让 Agent 能**直接操作 GUI**——看屏幕、移鼠标、点按钮、打字。这意味着理论上 Agent 能操作**任何软件**。

### 5.4 产品工程成熟

| 工程问题 | 解决方案 |
|---------|---------|
| Agent 跑飞了怎么办 | 人类确认机制（Human-in-the-loop） |
| Token 花费太高 | 上下文压缩、智能缓存 |
| 安全风险 | 沙箱执行、权限控制 |
| 结果不可预测 | 流式输出、实时展示 Agent 思考过程 |

## 六、Coding Agent 深入

Coding Agent 是目前最成功的 Agent 落地场景。以 Claude Code 为例：

### 6.1 Claude Code 的工作方式

```
用户: "帮我给这个函数加上错误处理"
  │
  ▼
[规划] 需要先读取函数代码 → 理解逻辑 → 添加 try-catch → 验证
  │
  ▼
[执行] Read(file.py) → 理解代码 → Edit(file.py, 添加错误处理) → Run(tests)
  │
  ▼
[反思] 测试通过了吗？→ 是 → 完成 / 否 → 分析错误 → 修改 → 重试
```

### 6.2 Coding Agent 的关键能力

| 能力 | 说明 |
|------|------|
| **代码理解** | 读取并理解整个项目的代码结构 |
| **精确编辑** | 不是重写整个文件，而是精确修改需要改的部分 |
| **终端执行** | 运行测试、安装依赖、执行构建等 |
| **多文件协调** | 修改涉及多个文件时保持一致性 |
| **自验证** | 修改后自动运行测试/编译确认 |

### 6.3 当前 Coding Agent 的局限

| 局限 | 说明 |
|------|------|
| **大规模重构** | 涉及几十个文件的重构仍然困难 |
| **理解业务逻辑** | 技术层面强，但理解复杂业务需求仍有差距 |
| **跨系统协调** | 涉及多个服务/系统的任务较难 |
| **创造性设计** | 能执行明确任务，但架构设计仍需人类主导 |

## 七、多 Agent 协作

单个 Agent 能力有限，多个 Agent 协作可以处理更复杂的任务：

```
┌───────────┐     ┌───────────┐     ┌───────────┐
│ 规划 Agent │ ──→ │ 编码 Agent │ ──→ │ 测试 Agent │
│ (分解任务) │     │ (写代码)   │     │ (验证正确) │
└───────────┘     └───────────┘     └───────────┘
      ↑                                    │
      └────────── 反馈修改 ◄───────────────┘
```

模式：
- **顺序流水线**：任务依次传递（规划 → 编码 → 测试）
- **平行分工**：多个 Agent 同时处理不同子任务
- **分层管理**：一个"主管"Agent 分配任务给"工人"Agent

## 八、Agent 评估

| 评估维度 | 指标 | 说明 |
|---------|------|------|
| **任务完成率** | 成功率 % | 给定任务能否正确完成 |
| **效率** | 步骤数 / Token 消耗 | 完成任务的代价 |
| **鲁棒性** | 遇到错误后的恢复能力 | 不崩溃、能自我纠正 |
| **安全性** | 有无危险操作 | 不删除重要文件、不泄露数据 |

常用 Benchmark：
- **SWE-bench**：真实 GitHub issue 修复（Coding Agent 标准评测）
- **WebArena**：网页操作任务
- **GAIA**：通用 AI Agent 评测

## 九、RAG vs Tool Calling vs Agent 总结

```
            能力递增 →
  ┌──────────┬─────────────┬──────────────┐
  │   RAG    │ Tool Calling │    Agent     │
  │          │              │              │
  │ 检索知识  │  调用工具     │ 自主规划执行  │
  │ 只读     │  单步调用     │ 多步循环     │
  │ 被动     │  按需触发     │ 主动         │
  └──────────┴─────────────┴──────────────┘

  Agent = LLM + 规划 + 记忆 + 工具(RAG + Tool Calling + ...)
```

三者是**包含关系**，不是替代关系。Agent 把 RAG 和 Tool Calling 作为自己的子能力来使用。
