# VLM 训练与评测

详解 VLM 的训练流程、数据构造方法、高效训练与部署技术。

## 一、训练流程总览

### 1.1 典型三阶段训练

```
阶段 1: 视觉-语言对齐预训练
├── 数据: 大规模图文对（数百万~数千万）
├── 训练: 只训投影层，冻结 ViT + LLM
├── 目标: 让视觉特征进入 LLM 的"语义空间"
├── 类比: "安装翻译器"
└── 耗时: 通常几小时到 1 天

阶段 2: 多任务预训练 (可选但推荐)
├── 数据: 多种视觉任务数据（描述、问答、OCR、定位...）
├── 训练: 投影层 + LLM（可选：解冻 ViT）
├── 目标: 学习多种视觉理解能力
├── 类比: "上学学各科"
└── 耗时: 通常数天到数周

阶段 3: 指令微调 / 偏好对齐
├── 数据: 高质量视觉指令数据 + 偏好数据
├── 训练: 全模型微调或 LoRA
├── 目标: 遵循指令、回答质量更好
├── 类比: "入职培训"
└── 耗时: 通常数小时到数天
```

### 1.2 各阶段的参数训练状态

| 阶段 | Vision Encoder | 投影模块 | LLM | 说明 |
|------|---------------|---------|-----|------|
| Stage 1 | 冻结 | **训练** | 冻结 | 只训连接器，最安全 |
| Stage 2 | 冻结/解冻 | **训练** | **训练** | 开始解锁 LLM |
| Stage 3 | 冻结/解冻 | **训练** | **训练** | 全量或 LoRA |

**为什么 Stage 1 要冻结 ViT 和 LLM？**

```
如果一开始就全部解冻:
  - 投影层是随机初始化的
  - 随机投影 → 给 LLM 喂的是"噪声"
  - LLM 被噪声梯度破坏 → 灾难性遗忘
  - ViT 也被不对齐的梯度破坏

所以先冻结两端，让投影层先学会"翻译"：
  ViT (固定好的视觉特征) → 投影层 (学习映射) → LLM (固定好的语义空间)

等投影层学好了，再逐步解冻其他组件。
```

## 二、训练数据详解

### 2.1 Stage 1: 图文对齐数据

| 数据集 | 规模 | 说明 |
|--------|------|------|
| **LAION-5B** | 50 亿 | 从网页爬取的图文对，规模最大但质量参差 |
| **LAION-COCO** | 6 亿 | LAION 子集，用 BLIP 重新生成 caption |
| **CC3M/CC12M** | 300 万/1200 万 | Google Conceptual Captions，质量较好 |
| **SBU** | 100 万 | 带 caption 的 Flickr 图片 |
| **ShareGPT4V** | 120 万 | 用 GPT-4V 生成的高质量描述 |

**LLaVA 的 Stage 1 数据**：CC3M + SBU 子集 = 595K 图文对，训练任务是图像描述（image captioning）。

### 2.2 Stage 2: 多任务预训练数据

```
任务类型          数据集示例                     规模
──────────────────────────────────────────────────
图像描述          COCO Captions                  500K+
视觉问答 (VQA)   VQAv2, GQA, OK-VQA            数百万
OCR/文档         TextVQA, DocVQA, ChartQA       100K+
视觉推理          NLVR2, Visual7W               200K+
视觉定位          RefCOCO, Visual Genome         数百万
知识问答          A-OKVQA, ScienceQA            50K+
```

### 2.3 Stage 3: 指令微调数据

**数据构造的核心方法**：

#### 方法一：GPT-4 辅助生成（LLaVA 方案）

```
流程:
1. 准备图像的元信息 (caption + bbox 标注)
2. 将元信息（纯文本）发给 GPT-4
3. GPT-4 生成多种指令-回答对:
   - 对话型: "图中有什么？" → 详细回答
   - 详细描述型: "请详细描述这张图" → 长文本描述
   - 复杂推理型: "这个场景说明了什么？" → 推理分析

示例:
  输入给 GPT-4: "图中有一只金毛犬(bbox: [100,200,400,500])在公园草地上"
  GPT-4 生成:
    Q: "这只狗在做什么？它可能在想什么？"
    A: "这只金毛寻回犬正在公园的绿色草地上奔跑，看起来非常开心。
        它可能正在享受户外活动的乐趣..."
```

#### 方法二：模型自举（Self-play / Self-improvement）

```
1. 用当前版本的 VLM 生成回答
2. 用更强的模型（如 GPT-4V）或人工评估质量
3. 筛选高质量回答作为新的训练数据
4. 重复迭代
```

#### 方法三：学术数据集转换

```
将已有的 CV 数据集转换为对话格式:

VQAv2 原始格式:
  image: xxx.jpg, question: "What color is the car?", answer: "red"

转换为 VLM 训练格式:
  <image>
  User: What color is the car?
  Assistant: The car is red.
```

### 2.4 偏好对齐数据 (RLHF / DPO)

```
数据格式:
  image + question + chosen_response + rejected_response

来源:
1. 人工标注偏好 (成本高但质量好)
2. AI 辅助标注:
   - 用 GPT-4V 判断两个回答哪个更好
   - 用规则过滤（如检查幻觉）

解决的问题:
  - 减少视觉幻觉（描述图中不存在的东西）
  - 提高回答的有用性和安全性
  - 对齐人类偏好
```

## 三、视觉幻觉问题

### 3.1 什么是视觉幻觉？

VLM 描述了图中**不存在**的内容：

```
示例:
  图片: 一只猫坐在桌子上

  VLM 回答: "图中有一只猫和一只狗坐在桌子上"
                              ^^^
                          幻觉！图中没有狗
```

### 3.2 幻觉的原因

```
1. 语言先验 (Language Prior):
   LLM 预训练时学到 "猫和狗" 经常共现
   → 看到猫就倾向于提到狗

2. 视觉信息不足:
   低分辨率或 token 压缩过度
   → VLM 看不清，只能靠猜

3. 训练数据偏差:
   训练数据中某些物体/场景更常见
   → 模型对常见物体有偏好

4. 描述过度 (Over-description):
   训练时鼓励详细描述
   → 模型为了"说更多"而编造内容
```

### 3.3 缓解幻觉的方法

| 方法 | 原理 | 代表 |
|------|------|------|
| **RLHF/DPO** | 用偏好数据惩罚幻觉回答 | LLaVA-RLHF |
| **高分辨率** | 看得更清楚，减少猜测 | InternVL2 |
| **对比学习微调** | 对比有/无幻觉的回答 | HACL |
| **推理时干预** | 采样时降低不确定 token 的概率 | VCD |
| **自我检查** | 生成后让模型自查是否有幻觉 | Woodpecker |

## 四、高效训练技术

### 4.1 LoRA 微调

当全量微调 LLM 成本太高时，使用 LoRA：

```
全量微调 70B LLM: 需要 ~320GB 显存 (约 4×A100-80G)
LoRA 微调 70B LLM: 需要 ~160GB 显存 (约 2×A100-80G)

LoRA 原理:
  在 LLM 的 attention 层插入低秩矩阵
  W' = W + ΔW = W + BA    (B: d×r, A: r×d, r << d)

  原始参数: 不动
  新增参数: 仅 r×d×2 ≈ 原参数的 0.1%~1%
```

### 4.2 混合精度与量化训练

```
常见配置:
  - ViT: FP16 / BF16
  - 投影层: FP16 / BF16 (参数少，不需要量化)
  - LLM: BF16 或 Int8 量化训练 (QLoRA)

QLoRA:
  LLM 权重用 4-bit 量化存储
  LoRA 部分用 BF16 训练
  → 大幅降低显存需求
```

### 4.3 数据并行与模型并行

```
小规模 VLM (≤13B):
  数据并行 (DDP) 即可
  每张 GPU 放一份完整模型

大规模 VLM (70B+):
  模型并行 (Tensor Parallel / Pipeline Parallel)
  将模型切分到多张 GPU

  常见配置:
  - InternVL2-76B: 8×A100-80G, Tensor Parallel=8
  - Qwen-VL2-72B: 类似配置
```

### 4.4 训练成本参考

| 模型 | 参数量 | 训练硬件 | 训练时间 | 估算成本 |
|------|--------|---------|---------|---------|
| LLaVA-1.5-13B | 13B | 8×A100 | ~1 天 | ~$500 |
| InternVL2-26B | 26B | 32×A100 | ~5 天 | ~$20K |
| InternVL2-76B | 76B | 128×A100 | ~2 周 | ~$200K+ |

## 五、从训练到部署

### 5.1 部署优化

```
1. 模型量化:
   FP16 → INT8 → INT4
   显存减半~四分之一，速度提升

2. KV Cache 优化:
   视觉 token 数多 → KV cache 大
   - PagedAttention (vLLM)
   - 视觉 token 的 KV cache 可以跨请求共享 (相同图片)

3. 视觉编码预计算:
   如果图片固定，ViT 编码结果可以缓存
   只需要运行投影层 + LLM

4. Speculative Decoding:
   用小模型预测，大模型验证
   在 VLM 上也有效
```

### 5.2 推理框架

| 框架 | 特点 | 适用场景 | 上手难度 |
|------|------|---------|---------|
| **vLLM** | PagedAttention，高吞吐，原生支持多种 VLM | 线上服务，高并发 | 中等 |
| **SGLang** | RadixAttention，适合多轮 VLM 对话 | 多轮对话、复杂 prompt 复用 | 中等 |
| **TensorRT-LLM** | NVIDIA 优化，延迟最低 | 追求极致延迟、已有 NVIDIA 部署体系 | 高 |
| **Ollama** | 简单易用，适合本地部署 | 本地试玩、个人使用 | 极低 |

**核心瓶颈**：LLM 推理时 KV Cache 占显存过多，长对话/多用户并发时显存很快耗尽，且预分配的连续空间利用率低。

#### vLLM — PagedAttention

```
核心思想: 像操作系统管理内存一样管理 KV Cache

传统做法:
  每个请求预分配一大块连续显存放 KV Cache
  → 请求短的浪费，请求长的可能不够，利用率约 50%

PagedAttention:
  把 KV Cache 切成小"页"，按需分配，不要求连续
  → 跟操作系统的虚拟内存分页机制一样
  → 显存利用率提升到约 95%
  → 同样的 GPU 能多服务 2-4 倍的并发请求
```

#### SGLang — RadixAttention

```
核心思想: 用前缀树缓存已计算的 KV Cache，避免重复计算

场景: 多轮对话每轮都带着完整历史
  第 1 轮: [系统提示 + 问题1] → 算完，KV Cache 存入 Radix Tree
  第 2 轮: [系统提示 + 问题1 + 回答1 + 问题2]
           ^^^^^^^^^^^^^^^^^^^^^^
           这部分 KV Cache 直接命中缓存，不用重算

→ 多轮对话和共享 system prompt 的场景加速明显
```

#### TensorRT-LLM — NVIDIA 底层优化

```
核心思想: 针对 NVIDIA GPU 做硬件级深度优化

三大手段:
  1. Kernel Fusion: 多个小操作合并为一个 GPU kernel，减少调度开销
  2. 量化加速: INT8/INT4 使用 Tensor Core 专用硬件，比软件模拟快很多
  3. 定制 CUDA Kernel: 每个算子针对特定 GPU 架构 (A100/H100) 手动调优

→ 单次推理延迟最低，但需要将模型转为 TensorRT 引擎格式，适配工作量大
```

#### Ollama — 一行命令跑模型

```
核心思想: 极致易用性，LLM 的 "Docker"

底层使用 llama.cpp (C++ 推理引擎)
  $ ollama run llama3    # 自动下载 + 量化 + 运行

→ 支持 CPU 推理，不一定需要 GPU
→ 性能一般，但谁都能跑，适合本地实验
```

> **实际使用**：生产环境首选 vLLM（生态好、模型支持广）；需要极致性能上 TensorRT-LLM；本地实验用 Ollama。
