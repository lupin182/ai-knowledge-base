# 视频 VLM 详解

系统讲解视频理解 VLM 的核心挑战、主流架构设计与关键技术。（内容更新至 2026.04）

## 一、视频 VLM 与图像 VLM 的核心差异

### 1.1 为什么视频理解更难？

图像 VLM 处理的是单帧静态信息，而视频 VLM 需要额外处理**时间维度**，带来三个核心挑战：

```
挑战 1: Token 数量爆炸
  图像: 1 帧 × 576 token = 576 token (LLaVA-1.5)
  视频: 32 帧 × 576 token = 18,432 token
  长视频: 128 帧 × 576 token = 73,728 token  ← 远超大多数 LLM 的有效窗口

挑战 2: 时序理解
  图像只需要理解"是什么"
  视频还需要理解"发生了什么"、"什么顺序"、"因果关系"
  例: "这个人先拿起杯子，然后喝水" → 需要跨帧推理

挑战 3: 信息冗余
  30fps 视频中，相邻帧 95%+ 像素几乎相同
  如何高效提取关键信息，丢弃冗余，是核心工程问题
```

### 1.2 视频 VLM 的通用流程

```
输入视频
    │
    ├── Step 1: 帧采样
    │   选择代表性帧（均匀采样 / 关键帧 / 自适应）
    │
    ├── Step 2: 逐帧视觉编码
    │   每帧独立通过 ViT → N 组 patch 特征
    │
    ├── Step 3: 时序建模（视频 VLM 的关键差异）
    │   建模帧与帧之间的时间关系
    │
    ├── Step 4: Token 压缩
    │   将海量视觉 token 压缩到 LLM 可承受的范围
    │
    └── Step 5: 送入 LLM
        压缩后的视频 token + 文本 query → 回答
```

### 1.3 时序建模的三种范式

视频 VLM 与图像 VLM 最本质的区别在于**如何处理帧与帧之间的时间关系**。目前主流方案可以归纳为三种范式，它们代表了"在哪个阶段、由谁来学时序"这个问题的三种不同回答：

```
直觉类比:
  范式 1 → 把一堆照片扔给一个聪明人，让他自己看出时间线
  范式 2 → 先请一个助手把照片按时间排好、标注关系，再交给聪明人
  范式 3 → 直接给一段录像，不拆成照片
```

#### 范式 1: 无显式时序建模（Early Fusion / Implicit Temporal Modeling）

**核心思想**：不设计任何专门的时序模块，让 LLM 自己通过 Self-Attention 学习帧间的时间关系。

```
架构流程:
  视频 → 采样 N 帧 → 逐帧独立通过 ViT 编码
       → 帧1 token | 帧2 token | ... | 帧N token  (直接拼接)
       → [可选] 每帧 token 压缩（池化/下采样）
       → 所有帧 token + 文本 token 一起送入 LLM
       → LLM 通过 Self-Attention 隐式学习"帧3和帧7之间发生了什么变化"

为什么 LLM 能学到时序关系？
  1. 位置编码: 帧1的token在前，帧N的token在后，LLM通过位置编码感知顺序
  2. Self-Attention: 每个token都能attend到所有其他帧的token
     → LLM 可以学到"这帧的手在桌上，下一帧手拿起了杯子"这种跨帧变化
  3. 足够的视频训练数据: 让 LLM 从数据中涌现出时序理解能力

代表模型:
  Video-LLaVA (2023.11):    8帧，LanguageBind 编码，无压缩
  LLaVA-OneVision (2024):   32~64帧，SigLIP 编码，Bilinear 池化压缩
  LLaVA-Video (2024.11):    同上，更好的训练数据
  InternVL2/2.5 (2024):     16~64帧，InternViT 编码，Pixel Shuffle 压缩
```

**优点**：
- **架构极其简洁**：不需要设计额外的时序模块，直接复用图像 VLM 架构
- **图像能力不退化**：图像就是"1帧的视频"，统一处理
- **工程友好**：训练和推理流程与图像 VLM 几乎一致
- **实践效果最好**：2024 年后，这类方法在多数基准上反而优于显式时序模块

**缺点**：
- **LLM 计算负担重**：所有帧的 token 拼接后很长，Self-Attention 是 $O(n^2)$ 复杂度
- **长视频困难**：32帧 × 256 token = 8192 token 已经很多，128帧就接近 LLM 上下文上限
- **时序理解是"副产品"**：LLM 本质是为语言设计的，时序理解需要足够的视频数据才能涌现
- **没有时序归纳偏置**：模型不知道"相邻帧之间的关系比远距离帧更密切"这种先验

**为什么它反而赢了？** 2024 年后的实践反复证明，只要图像 VLM 足够强 + 视频训练数据足够多，LLM 的 Self-Attention 足以隐式学到时序关系。这与 NLP 领域的趋势一致——**通用的大模型 + 足够的数据 > 精心设计的归纳偏置**。

#### 范式 2: 显式时序模块（Temporal Module / Late Fusion）

**核心思想**：在 ViT 编码之后、LLM 之前，插入一个专门的时序建模模块，先把时间关系学好，再把融合后的表示交给 LLM。

```
架构流程:
  视频 → 采样 N 帧 → 逐帧独立通过 ViT 编码
       → N 组帧级特征
       → ┌──────────── 时序模块（关键差异）────────────┐
         │  方案 A: 时空池化                            │
         │    空间池化: 每帧的 patch 取平均 → 帧级向量   │
         │    时间池化: 跨帧对每个 patch 取平均          │
         │    代表: Video-ChatGPT                       │
         │                                             │
         │  方案 B: 时序 Q-Former                       │
         │    用一组可学习的 query token                 │
         │    通过 Cross-Attention 从所有帧中提取信息    │
         │    query 数量固定 → 自然实现 token 压缩       │
         │    代表: VideoChat, VideoChat2               │
         │                                             │
         │  方案 C: 自适应池化                          │
         │    对帧维度做 Adaptive Average Pooling        │
         │    16帧压缩到2~4帧的token量                  │
         │    代表: PLLaVA                              │
         │                                             │
         │  方案 D: 时序 Transformer                    │
         │    在 patch 级别跨帧做 Temporal Attention     │
         │    每个空间位置的 token 在时间维度上交互       │
         │    代表: TimeSformer 思路的变体               │
         └──────────────────────────────────────────────┘
       → 压缩后的视频 token → 投影 → LLM

具体例子 — VideoChat2 的 Q-Former:
  输入: 16帧 × 每帧 257 token × 1024 维
  Q-Former 内部:
    32 个可学习 query (每个 768 维)
    ↓ Cross-Attention
    query attend 到所有 16 帧的所有 patch token
    query 自动学会: "我关注动作变化" / "我关注场景切换" 等
    ↓
  输出: 32 个 token × 768 维
  → 16帧 × 257 token = 4112 token 压缩成 32 token！
```

**优点**：
- **时序建模更可控**：显式设计模块来学习时间关系，有更强的归纳偏置
- **天然做 token 压缩**：时序模块输出固定数量的 token，减轻 LLM 负担
- **LLM 负担小**：LLM 只需处理压缩后的少量 token

**缺点**：
- **额外模块增加复杂度**：需要单独设计、训练时序模块
- **信息瓶颈**：压缩过程中可能丢失关键细节（如某一帧中的小物体）
- **训练更复杂**：通常需要多阶段训练（先训时序模块，再微调整体）
- **实际性能不如范式1**：2024年后的实验表明，复杂的时序模块并没有带来显著优势

**为什么它逐渐式微？** Q-Former 等模块虽然优雅，但存在"信息瓶颈"问题——将数千个 token 压缩到几十个，不可避免地丢失细节。而随着 LLM 上下文越来越长（32K → 128K → 1M），"让 LLM 直接看更多 token"反而比"先压缩再看"效果更好。

#### 范式 3: 时空联合编码（Spatiotemporal Encoding）

**核心思想**：不把视频拆成一帧帧的图像分别编码，而是直接把视频当作**3D 信号**（时间 × 高度 × 宽度）整体输入编码器。

```
架构流程:
  视频片段 (T帧 × H × W × 3)
       → 3D Patch Embedding:
         不是 16×16 的 2D patch，而是 2×16×16 的 3D patch
         即每个 patch 跨越 2 帧 + 16×16 空间区域
         → 每个 patch 天然包含时序信息
       → 时空 Transformer:
         每个 token 同时 attend 到空间和时间邻居
         不再区分"空间 attention"和"时间 attention"
       → 视频级特征
       → 投影 → LLM

代表模型:
  InternVideo2: 使用 VideoMAE 式的时空编码器
    VideoMAE 预训练: 随机遮蔽 90% 的时空 patch → 重建
    → 编码器被迫学习时空连续性

  Qwen2-VL 的 3D 卷积 merge (部分属于此范式):
    每 2帧 × 2×2 空间 patch → 合并为 1 个 token
    虽然 ViT 主体仍是逐帧编码，但 merge 操作引入了时空联合

  3D-RoPE (Qwen2-VL/2.5-VL):
    为每个 token 编码 (t, h, w) 三维坐标
    → 模型从位置编码层面就区分时空关系
    例: token A 在 (t=3, h=5, w=8)，token B 在 (t=7, h=5, w=8)
        → 模型知道它们在同一空间位置但相隔4帧（同一物体的运动轨迹）
```

**优点**：
- **时空信息联合建模**：不存在"先空间后时间"的信息割裂
- **更好捕捉运动信息**：3D patch 天然编码了局部运动（光流类信息）
- **理论上更优**：视频本质就是 3D 信号，3D 编码更符合数据本质

**缺点**：
- **无法复用图像预训练**：图像 ViT（如 CLIP、SigLIP）只做过 2D 训练，权重不能直接迁移到 3D 编码器
- **预训练成本高**：需要大规模视频数据从头训练 3D 编码器
- **图像能力受限**：针对视频设计的编码器处理单张图像时可能不如专门的图像编码器

#### 三种范式的对比总结

```
                    范式 1              范式 2               范式 3
                  (隐式/LLM学)       (显式时序模块)        (时空联合编码)
─────────────────────────────────────────────────────────────────────────
时序学习位置      LLM 内部            ViT 和 LLM 之间       视觉编码器内部
架构复杂度        最低                中等                  最高
图像 ViT 复用     ✅ 直接复用          ✅ 直接复用           ❌ 需要重新训练
Token 数          多（所有帧拼接）     少（模块压缩）        中等
训练难度          低                  中（多阶段）          高（需大量视频预训练）
长视频扩展性      较差（token 线性增长）较好（压缩后固定）    取决于实现
2024-25 主流度    ⭐⭐⭐ 主流           ⭐ 逐渐减少           ⭐⭐ 特定场景
代表              LLaVA-OV, InternVL  VideoChat2, PLLaVA    InternVideo2, Qwen2-VL(部分)

趋势结论:
  范式 1 成为绝对主流 → "让 LLM 学一切"的思路胜出
  范式 2 的设计思想被简化吸收（如简单池化替代 Q-Former）
  范式 3 的思想部分融入范式 1（如 Qwen2-VL 的 3D 卷积 merge + 3D-RoPE）
  → 最终趋势: 三种范式的边界正在模糊化，取长补短
```

### 1.4 Token 爆炸：视频 VLM 的第一工程问题

范式 1 成为主流后，所有帧的 token 直接拼接送入 LLM，**token 数量管理成为视频 VLM 最核心的工程挑战**。解决思路可以归纳为三个方向：从时间维度减少帧数、从空间维度压缩每帧 token、从 LLM 侧扩大容量。

```
Token 数 = 采样帧数 × 每帧 token 数

  原始规模: 128 帧 × 576 token/帧 = 73,728 token  ← 远超 LLM 有效窗口
  优化目标: 在信息损失最小的前提下，把 token 数压到 LLM 能高效处理的范围

  三个优化方向:
  ┌─────────────────────────────────────────────────────────┐
  │  方向 1: 时间维度 — 少采帧（减少"采样帧数"）             │
  │  方向 2: 空间维度 — 压缩每帧 token（减少"每帧 token 数"）│
  │  方向 3: LLM 侧 — 扩大上下文 + 高效注意力               │
  └─────────────────────────────────────────────────────────┘
```

#### 方向 1: 帧采样策略（时间维度压缩）

采样没采到的帧，对模型来说就是**不存在的**。当没有显式时序模块时，帧采样策略直接决定了模型能"看到"什么信息。

```
主流采样方式:

  1. 均匀采样 (Uniform Sampling):
     将视频等间隔取 N 帧（如 32 帧）
     优点: 简单，覆盖全程
     缺点: 不区分信息密度，可能在无意义片段浪费帧预算
     代表: LLaVA-OneVision, InternVL2

  2. 固定帧率采样 (Fixed FPS):
     按固定帧率取帧（如 2fps）
     优点: token 数可预测，与视频时长线性相关
     缺点: 快速运动场景欠采样，静态场景过采样
     代表: Qwen2-VL (2fps)

  3. 动态帧率采样 (Dynamic FPS):
     根据视频内容自适应调整采样帧率
     快速运动 → 高帧率采样; 静态场景 → 低帧率采样
     代表: Qwen2.5-VL, Qwen3-VL

  4. 镜头感知采样 (Shot-Aware Sampling):
     先将视频按镜头切换分段，每段选取代表帧
     例: InfoShot 从每个镜头中选 2 帧 — 一帧代表主体，一帧捕捉变化
     优点: 不遗漏场景切换中的关键信息
     代表: InfoShot (2026)

  5. 查询驱动采样 (Query-Based Sampling):
     根据用户问题动态选取最相关的帧
     例: Frame-Voyager 学习根据文本 query 选取最有信息量的帧组合
     优点: 同等帧数下信息密度最高
     缺点: 需要额外的帧选择模块，增加推理延迟
     代表: Frame-Voyager (ICLR 2025), EFS (2025)
```

**关键研究发现**（Frame Sampling Strategies Matter, 2025）：
- 在相同模型上，不同采样策略可导致 **10%+** 的性能差异
- 当前视频基准存在严重的**采样偏差**——不同模型用不同采样策略评测，无法公平比较
- 未来趋势：将帧选择集成到模型训练中，让模型自己学习最优采样策略

#### 方向 2: 每帧 Token 压缩（空间维度压缩）

即使采样帧数固定，每帧产生的 token 数仍然很多（ViT 输出通常 576~1024 token/帧）。主流方案在 ViT 编码之后、送入 LLM 之前，用**轻量级操作**压缩每帧 token：

```
方案对比:

  1. Bilinear 池化 (LLaVA-OneVision, LLaVA-Video):
     对每帧的 patch 特征做 2×2 空间下采样
     729 token/帧 → ~196 token/帧（压缩 3.7x）
     无额外参数，纯几何操作

  2. Pixel Shuffle (InternVL2/2.5/3):
     将相邻 2×2 个 patch token 合并为 1 个（通道维度拼接后投影）
     1024 token/帧 → 256 token/帧（压缩 4x）
     少量投影参数，保留更多信息

  3. 3D 卷积 Merge (Qwen2-VL/2.5-VL/3-VL):
     同时在时间和空间维度压缩: 每 2帧 × 2×2 空间 patch → 1 个 token
     → 一次操作同时做了时间压缩和空间压缩（压缩 8x）
     这也是为什么 Qwen2-VL 被归为"部分属于范式 3"

  4. 动态压缩 — ViCO + ViR (InternVL3.5, 2025):
     不再一刀切，而是根据内容动态决定压缩率:
     ├── ViCO (Visual Consistency Learning):
     │   训练模型在不同压缩率下给出一致的回答
     │   → 模型学会对压缩不敏感
     └── ViR (Visual Resolution Router):
         动态选择每帧的最佳 token 数量
         简单场景 → 64 token/帧; 复杂场景 → 256 token/帧
     效果: 同等质量下推理加速最高 4.05x

具体数字对比（32 帧视频）:
  无压缩:        32 × 729  = 23,328 token
  Bilinear:      32 × 196  =  6,272 token  ← LLaVA-OV
  Pixel Shuffle: 32 × 256  =  8,192 token  ← InternVL2
  3D Conv Merge: ~16 × 182 =  2,912 token  ← Qwen2-VL (时间也压了)
  动态(ViR):     32 × 64~256 = 2,048~8,192 token ← InternVL3.5
```

**关键区别**：这些压缩方法与范式 2 的 Q-Former 等时序模块**本质不同**——它们只做简单的空间/时空下采样，不试图"学习时序关系"。信息虽然有损失，但损失可控且均匀，不像 Q-Former 那样将数千 token 压缩到几十个造成严重信息瓶颈。

#### 方向 3: LLM 侧扩容

既然视频 token 多，另一个思路是让 LLM 能处理更多 token：

```
上下文长度扩展:
  2024: 32K → 128K（大部分模型）
  2025: 256K（Qwen3-VL 原生支持，可扩展至 1M）
  → 能容纳更多帧，减少采样时的信息丢失

高效注意力机制:
  窗口注意力 + 全局注意力混合 (Qwen2.5-VL):
    ViT 内部用 window attention 降低计算量
    周期性插入 full attention 层保持全局信息流通
    → 支持更长视频输入而不爆显存

  解耦部署 DvD (InternVL3.5):
    视觉编码器和语言模型分别部署在不同 GPU 上
    视觉处理和语言处理并行
    → 提升推理吞吐量
```

#### Token 管理策略总结

```
              时间维度              空间维度             LLM 侧
            （少采帧）           （压缩每帧）          （扩容量）
──────────────────────────────────────────────────────────────────
简单方案    均匀采样 32帧        Bilinear 2×2 池化     扩展上下文 128K
中等方案    固定 fps 采样        Pixel Shuffle 4x      窗口注意力
高级方案    动态 fps / 查询驱动   3D 卷积 merge 8x      解耦部署
最新方案    模型自学最优采样      动态压缩 ViCO+ViR     1M 上下文

实际做法: 三个方向通常组合使用
  例 Qwen2.5-VL: 动态 fps + 3D 卷积 merge + 窗口注意力 → 支持小时级视频
  例 InternVL3.5: 均匀采样 + 动态压缩 ViR + 解耦部署 → 4x 推理加速
```

## 二、主流视频 VLM 架构

### 2.1 Video-ChatGPT (2023.06)

**最早的视频指令微调模型之一**，架构非常清晰：

```
架构:
  视频 → 均匀采样 N 帧 → CLIP ViT-L/14 逐帧编码
       → 时空池化（Spatial + Temporal Pooling）→ 压缩
       → Linear 投影 → 视觉 token
       + 文本 token → Vicuna-7B → 回答

时空池化细节:
  输入: N 帧 × T 个 patch token × D 维
  ├── 空间池化: 对每帧的 T 个 patch 取平均 → N×D → N 个时序 token
  └── 时间池化: 对每个 patch 跨帧取平均 → T×D → T 个空间 token
  拼接: N + T 个 token → 送入 LLM
```

**核心贡献**：
- 构建了 100K 视频指令数据（用 GPT-3.5 基于视频 caption 生成）
- 提出视频对话评测框架（正确性、细节、上下文、时序、一致性 5 个维度）

**局限**：时空池化过于简单，丢失了大量细节信息。

### 2.2 VideoChat / VideoChat2 (2023.05 / 2023.11)

```
VideoChat (v1):
  视频 → 逐帧 ViT 编码 → Q-Former（类 BLIP-2）→ 压缩为固定数量 token
       → 拼接文本 → Vicuna-7B → 回答
  特点: 借鉴 BLIP-2 的 Q-Former 做视频 token 压缩

VideoChat2 (v2) 关键改进:
  1. 统一图像和视频训练:
     ├── 图像数据: 用于学习空间理解
     └── 视频数据: 用于学习时序理解
     → 图像和视频共享同一架构，不分开训练

  2. 渐进式训练:
     Stage 1: 图文对齐（冻结 ViT + LLM，训 Q-Former）
     Stage 2: 图像指令微调
     Stage 3: 视频指令微调
     → 先学空间理解，再学时序理解

  3. 更强的视觉编码器:
     从 CLIP ViT-L → UMT-L（专门做视频理解的编码器）
```

### 2.3 Video-LLaVA (2023.11)

**最简洁的视频 VLM 之一**，核心思想：把图像和视频统一到同一个表示空间。

```
架构:
  图像 → ViT → MLP → 视觉 token ──┐
                                    ├── 共享同一个 LLM → 回答
  视频 → 均匀采样 8 帧              │
       → LanguageBind Video Encoder │
       → MLP → 视频 token ─────────┘

关键设计: LanguageBind
  不用 CLIP ViT 编码视频，而是用 LanguageBind 的视频编码器
  LanguageBind 将图像、视频、音频等模态都对齐到语言空间
  → 图像和视频的 token 天然在同一语义空间，LLM 可以统一处理
```

**训练策略**：
1. Stage 1: 图文 + 视频文本对齐（训投影层）
2. Stage 2: 图像 + 视频指令混合微调（训投影层 + LLM）

### 2.4 LLaVA-OneVision (2024)

**目前最主流的统一图像/视频方案之一**，直接将 LLaVA-NeXT 扩展到视频：

```
视频处理流程:
  视频 → 均匀采样 N 帧（如 32 帧）
       → 每帧独立通过 SigLIP 编码 → 每帧 729 token
       → 2×2 Bilinear Interpolation 池化 → 每帧约 196 token
       → 所有帧 token 拼接 + 特殊的 <video> token 分隔
       → MLP 投影 → 送入 Qwen2 LLM

token 数管理:
  32 帧 × 196 token = 6272 token（可接受）
  支持最多约 64 帧

关键: 没有显式时序模块
  完全依赖 LLM 的长上下文 attention 来捕捉时序关系
  实践证明: 足够好的图像 VLM + 足够多的视频数据 = 强视频理解能力
```

**训练策略（三阶段）**：
```
Stage 1: 单图训练
├── 大规模图文对齐 + 图像指令微调
└── 建立强大的空间理解基础

Stage 2: 单图高质量微调
├── 高质量图像指令数据
└── 提升单图推理能力

Stage 3: OneVision 混合微调
├── 单图 + 多图 + 视频 数据混合训练
├── 视频数据包括: 视频问答、视频描述、时序推理等
└── 关键发现: 单图能力不会因为加入视频数据而退化
```

### 2.5 PLLaVA (2024)

**Pooling LLaVA**，用简单的自适应池化解决视频 token 过多问题：

```
架构:
  视频 → 采样 16 帧 → 逐帧 CLIP ViT 编码
       → Adaptive Average Pooling (跨帧)
       → 将 16 帧的 token 池化为等效于 1~4 帧的 token 数量
       → MLP 投影 → LLM

池化策略:
  输入: [16, H, W, D]（16帧，每帧 H×W 个 token）
  Adaptive Pool 沿时间维度: [16, H, W, D] → [K, H, W, D]
  K 通常取 2 或 4，即把 16 帧压缩到 2~4 "帧"
  → 大幅减少 token 数，同时保留关键时序信息
```

**核心发现**：简单的池化效果意外地好，不需要复杂的时序模块。

### 2.6 InternVL2（视频模式）

InternVL2 天然支持视频输入，处理方式与图像类似：

```
视频 → 均匀采样 N 帧
     → 每帧当作一个"tile"，复用图像处理流程:
        每帧 → InternViT-6B → 1024 token → Pixel Shuffle → 256 token
     → 所有帧 token 拼接 + 全局缩略图 token
     → MLP 投影 → InternLM2 → 回答

Token 数:
  16 帧: 16 × 256 = 4096 token
  32 帧: 32 × 256 = 8192 token
  → 受限于 LLM 上下文长度，通常限制在 16~32 帧
```

### 2.7 Qwen2-VL（视频模式）

Qwen2-VL 对视频有独特的处理方式：

```
视频 → 按 2fps 采样帧（不是固定帧数，而是固定帧率）
     → 每帧按原始分辨率送入 ViT（Naive Dynamic Resolution）
     → 3D 卷积 merge: 在时间和空间维度同时下采样
        具体: 每 2×2 空间 patch × 2 帧合并为 1 个 token
     → 大幅减少 token 数
     → 用 3D-RoPE（时间 + 高度 + 宽度 三个维度的旋转位置编码）
     → MLP 投影 → Qwen2 LLM

关键设计:
  1. 2fps 采样: 对于大部分视频足够，且 token 数可预测
  2. 3D 卷积 merge: 同时利用时间和空间冗余，比单独压缩更高效
  3. 3D-RoPE: 让模型感知每个 token 的时空位置
     → 传统 1D 位置编码无法区分"第3帧左上角"和"第5帧右下角"
     → 3D-RoPE 为每个 token 编码 (t, h, w) 三个坐标
```

### 2.8 Qwen2.5-VL (2025.01)

Qwen2-VL 的重大升级，**视频理解能力大幅提升**，尤其在长视频和时序定位方面：

```
架构改进（相比 Qwen2-VL）:
  1. 动态 FPS 采样:
     Qwen2-VL: 固定 2fps
     Qwen2.5-VL: 根据视频内容自适应调整采样帧率
     → 快速运动场景用更高帧率，静态场景用更低帧率
     → 同等 token 预算下捕获更多关键信息

  2. 窗口注意力 + 全局注意力混合:
     ViT 内部采用 window attention 降低计算量
     周期性插入 full attention 层保持全局信息流通
     → 支持更长视频输入而不爆显存

  3. 时序定位（Temporal Grounding）:
     新增能力: 给定问题，返回视频中相关片段的起止时间
     例: "这个人什么时候开始跑步？" → "00:03:12 - 00:03:45"
     → 不只是理解视频，还能精确定位

关键提升:
  - 支持超过 1 小时的视频理解
  - Video-MME 长视频子集上大幅领先 Qwen2-VL
  - 新增视频时序定位和事件检索能力
  - MLP 投影 → Qwen2.5 LLM（更强的基座）
```

### 2.11 原生多模态视频模型 (2025-2026)

2025 年起，**闭源模型率先实现了原生视频理解**，不再依赖"逐帧图像编码"的范式：

```
Gemini 2.0 / 2.5 (Google, 2025):
  - 原生支持视频输入，无需手动采样帧
  - 直接上传视频文件，模型内部处理时序
  - 支持超长视频（1 小时+），1M token 上下文
  - 视频+音频联合理解（不只看画面，还听声音）
  - 时间戳级别的精确问答

GPT-4o / GPT-4.1 (OpenAI, 2024-2025):
  - 原生多模态架构，视频作为一等公民输入
  - 支持视频帧 + 音频的联合推理
  - 在 Video-MME 等基准上表现强劲

趋势: 从"图像 VLM 的扩展"到"原生视频理解"
  早期: 视频 = 多张图片 → 逐帧编码 → 拼接
  现在: 视频作为独立模态，端到端处理
  → 闭源模型领先，开源模型正在追赶
```

#### "原生多模态"到底改了什么？

帧采样/压缩仍然存在，并没有什么魔法——本质区别在于**"谁来做、在哪里做、怎么训"**发生了根本变化。

**传统做法 vs 原生做法的核心对比**：

```
传统做法（图像 VLM 扩展到视频）:
  人为设定采样规则（均匀取32帧 / 固定2fps）
      ↓
  复用图像 ViT 逐帧编码（CLIP/SigLIP，本来是为单张图片训的）
      ↓
  人为设计拼接/压缩策略（池化、Q-Former 等）
      ↓
  塞进 LLM
  问题：每一步都是人为拼凑的，ViT 从没见过"帧与帧之间的关系"

原生多模态做法:
  视频作为连续信号输入（不是人手动抽帧）
      ↓
  模型内部的 video tokenizer 自己决定怎么"采样+编码"
  （预训练时就见过大量视频，学会了什么该保留、什么该丢弃）
      ↓
  时序压缩是学出来的，不是人为规定的
      ↓
  统一的多模态 backbone 处理
```

**三个关键变化**：

**变化 1：Tokenizer 层面 — 从复用图像 ViT → 视频原生 Tokenizer**

```
传统:
  CLIP ViT（只在图片上训过）→ 逐帧独立编码 → 帧之间没有任何交互
  本质上是把视频当成"一堆无关的图片"

原生:
  时空 Tokenizer 从预训练就处理视频 chunk
  例如 3D patch embedding: 把 (2帧×16×16) 作为一个 token
  → tokenizer 本身就编码了局部运动信息

  Gemini 的做法（根据公开信息推测）:
  视频 → 内部按 chunk 处理（如每 1-2 秒一个 chunk）
       → 每个 chunk 用时空编码器提取特征
       → 编码器在预训练时就见过海量视频
       → 自动学会: 静态场景产生少量 token，动态场景产生更多 token

所以帧采样仍然存在，但从"人为规则"变成了"模型学出来的"。
```

**变化 2：预训练层面 — 从"后加"→"一开始就有"**

```
传统路线（两步走）:
  Step 1: 在图文数据上预训练 → 学会空间理解
  Step 2: 在视频数据上微调 → 试图补上时序理解
  问题: 视频理解是"打补丁"加上去的，模型骨子里还是图像模型

原生路线（一步到位）:
  预训练数据 = 图文 + 视频文本 + 音频 混合
  从第一个 gradient 开始就学习:
    - 图片里"有什么"
    - 视频里"发生了什么"
    - 音频里"说了什么"
  → 时序理解是模型的"母语"，不是后学的"外语"

这就是 InternVL3 说的"原生多模态预训练"的意思。
```

**变化 3：上下文管理 — 从"全部塞进去"→"智能检索"**

```
传统:
  取 N 帧 → 全部编码 → 全部拼接 → 塞进 LLM 上下文
  受限于上下文长度，N 不能太大

Gemini 等原生模型（推测的做法）:
  1小时视频 → 全部编码并缓存（离线/异步）
  用户提问时 → 模型内部做"注意力路由":
    "用户问的是第30分钟的事"
    → 重点关注第28-32分钟的 token，其余帧极度压缩
  类似于 检索增强生成(RAG) 的思路用在了视频 token 上

  这解释了为什么 Gemini 能处理 1小时+ 视频:
  不是真的把 180万 token 全放进 attention
  而是分层处理 + 按需检索
```

**传统 vs 原生的总结对比**：

| 维度 | 传统视频 VLM | 原生多模态 |
|------|-------------|-----------|
| 帧采样 | 人为规则（均匀/fps） | 模型自己学怎么采 |
| 视觉编码器 | 图像 ViT（没见过视频） | 时空编码器（预训练见过） |
| 时序能力来源 | 微调时后加的 | 预训练时就学的 |
| 长视频处理 | 硬塞 / 暴力压缩 | 分层缓存 + 按需检索 |
| 信息损失 | 损失大且不可控 | 损失小且自适应 |

> **一句话总结**：不是不采样，是让模型自己学怎么采样和压缩，而且从预训练开始就教它理解视频，不是事后补课。

> ⚠️ **注意**：Gemini、GPT-4o 等闭源模型的具体实现细节并未完全公开，上面关于"智能检索""自适应采样"的部分是基于公开论文和技术博客的合理推测。开源社区中 InternVL3 的"原生多模态预训练"是目前最接近这个方向的公开工作。（截至 2026.04）

### 2.12 Qwen3-VL (2025.11)

Qwen 系列视觉语言模型的又一次重大升级，**在时空建模和长视频推理方面实现质的飞跃**：

```
架构改进（相比 Qwen2.5-VL）:

  1. 增强版 Interleaved-MRoPE:
     Qwen2.5-VL 的 3D-RoPE 基础上进一步升级

  2. DeepStack 集成:

  3. 文本时间戳对齐 (Text-Timestamp Alignment):

模型规格:
  Dense 版本: 2B / 4B / 8B / 32B

  原生支持 256K token 上下文，可扩展至 1M token
```

**关键提升**：
- 更好的视频 OCR（识别视频中的文字）
- 长视频理解能力大幅提升
- 视频时序定位（Temporal Grounding）更精确

### 2.13 InternVL3.5 (2025.08)

InternVL 系列的最新迭代，**核心创新在于推理效率和视觉 token 压缩**：

```
关键升级（相比 InternVL3）:

  1. 视觉一致性学习 (Visual Consistency Learning, ViCO):
     核心问题: 压缩视觉 token 后，模型回答质量下降
     ViCO 的做法:
       训练模型在不同压缩率下给出一致的回答
       例: 256 token/帧 和 64 token/帧 应该回答相同
     两阶段过程:
       Stage 1 (一致性训练): 教模型对压缩不敏感
       Stage 2 (路由器训练): 训练动态选择最佳压缩率
     → 同等质量下，推理速度最高 4.05x 加速

  2. 视觉分辨率路由器 (Visual Resolution Router, ViR):
     动态选择每张图/每帧的最佳 token 数量
     简单场景 → 少量 token; 复杂场景 → 更多 token
     → 避免"一刀切"的固定压缩率

  3. 解耦视觉-语言部署 (Decoupled Vision-Language Deployment, DvD):
     视觉编码器和语言模型分别部署在不同 GPU 上
     视觉处理和语言处理可以并行
     → 进一步提升推理吞吐量

视觉编码器: InternViT-300M / InternViT-6B
语言模型: 基于 Qwen3 / GPT-OSS 初始化
视频评测: 在 16/32/48/64 帧设置下均有提升
```

### 2.14 Qwen3.5-Omni — 全模态模型 (2026.03)

**首个真正意义上的开源全模态模型**，统一处理文本、图像、视频、音频，并支持实时语音生成：

```
架构: Thinker-Talker 双模块设计

  Thinker（思考器）:
    负责多模态理解和推理
    输入: 文本 + 图像 + 视频 + 音频 → 统一处理
    内部: Hybrid-Attention MoE（混合注意力专家混合）
    所有模态共享同一个推理引擎

  Talker（表达器）:
    负责生成文本和语音输出
    支持实时流式语音生成
    ARIA 技术: 自适应调节文本和音频的输出节奏
    → 语音输出自然流畅，无卡顿

视频处理:
  支持 400+ 秒 720p 视频 @ 1fps
  256K token 上下文

音频处理（关键新能力）:
  原生 Audio Transformer (AuT) 编码器
  在 1 亿+ 小时音视频数据上预训练
  → 不是"先 ASR 转文字再理解"，而是直接理解音频信号
  支持 10+ 小时音频输入
  语音识别: 113 种语言/方言
  语音生成: 36 种语言/方言

模型规格:
  30B MoE（每 token 激活 3B 参数）

为什么重要:
  这是第一个开源的"音视频联合理解"模型
  之前只有 Gemini 等闭源模型能做到
  → 开源社区在全模态方向上的重大突破
```

### 2.15 StreamingVLM — 实时流式视频理解 (2025.10)

**首个实现无限长度视频流实时理解的开源方案**（MIT Han Lab）：

```
核心问题:
  传统视频 VLM: 先录完视频 → 再分析
  实际需求: 视频流源源不断，需要实时理解（监控、直播、AR）
  挑战: KV Cache 随时间线性增长 → 内存和延迟爆炸

关键设计: 固定大小的 KV Cache

  KV Cache 三部分:
  ├── Attention Sinks: 保留最初几个 token 的 KV（稳定注意力分布）
  ├── 短窗口 (Recent Vision): 最近几秒的视觉 token（细节）
  └── 长窗口 (Recent Text): 最近的文本 token（对话上下文）

  新帧进入 → 编码 → 更新短窗口（丢弃最旧的视觉 token）
  → KV Cache 大小恒定，无论视频多长

训练策略:
  在短视频 chunk 上用全注意力做 SFT
  chunk 之间有重叠 → 模拟推理时的滑动窗口模式
  → 不需要在超长视频上训练，但推理时能处理无限长视频

性能:
  单张 H100 上实现 8 FPS 实时处理
  对 GPT-4o mini 的胜率达 66.18%
  评测基准: Inf-Streams-Eval（平均 2 小时以上的视频）
```

## 三、VideoChat-Flash 与长视频理解

### 3.1 长视频的核心困难

```
问题量化:
  1 小时视频, 2fps = 7200 帧
  每帧 256 token = 1,843,200 token
  → 即使 128K 上下文的 LLM 也远远不够

根本矛盾:
  信息量: 长视频包含大量信息，不能简单丢弃
  算力限制: LLM 的上下文窗口和计算成本有限
  → 必须在信息保留和计算成本之间找到平衡
```

### 3.2 长视频处理策略

```
策略 1: 更激进的采样
  10 分钟视频 → 只取 32~64 帧
  优点: 简单有效
  缺点: 可能错过关键瞬间（如"第3分12秒发生了什么？"）

策略 2: 分段处理 + 记忆机制
  将长视频切成多个短片段
  每段独立理解 → 生成文字摘要 / 压缩表示
  最终: 所有段的摘要 + 用户问题 → LLM 回答
  代表: MovieChat, MA-LMM

策略 3: 层级式理解
  第一层: 粗粒度全局理解（大间隔采样）
  第二层: 根据问题定位相关片段
  第三层: 对相关片段细粒度理解（密集采样）
  → 类似人类看视频的方式：先快进浏览，再对感兴趣的部分仔细看

策略 4: 流式处理
  不是一次性处理整个视频
  而是逐段输入，维护一个固定大小的"记忆缓冲区"
  新帧进来时，压缩或替换旧的记忆
  代表: VideoChat-Flash
```

### 3.3 VideoChat-Flash 的设计

```
核心思路: 将视频 token 组织为两层记忆

短期记忆 (Recent Buffer):
  最近 K 帧的完整视觉 token
  保留细节信息，用于回答关于"刚才发生了什么"的问题

长期记忆 (Compressed Memory):
  更早的帧经过压缩（时序合并 + 关键帧选择）
  只保留关键信息，大幅减少 token 数

信息流:
  新帧进入 → 加入短期记忆
  短期记忆满了 → 最旧的帧压缩后移入长期记忆
  回答问题时 → 长期记忆 + 短期记忆 + 文本 query → LLM

效果: 可以处理数小时的视频，同时保持对近期事件的细节理解
```

## 四、视频推理（Video Reasoning）

2025 年，受 LLM 推理（如 DeepSeek-R1、OpenAI o1）的成功启发，**将强化学习（RL）引入视频 VLM** 成为重要新方向。核心思路：不只让模型"描述所见"，还要让它"看到后思考推理"。

### 4.1 Video-R1 (2025.03, NeurIPS 2025)

**首个系统性探索 R1 范式用于视频推理的工作**：

```
核心问题:
  直接把 DeepSeek-R1 的 GRPO 算法用于视频 → 效果不好
  原因 1: 缺少时序建模的奖励信号 → 模型不利用帧间时序信息
  原因 2: 高质量视频推理数据稀缺

解决方案:

  1. T-GRPO (Temporal GRPO) 算法:
     在标准 GRPO 奖励基础上，增加"时序利用"奖励
     鼓励模型在推理过程中显式引用不同时间点的信息
     → 模型学会: "在第3秒看到X，在第8秒看到Y，因此推断Z"

  2. 两个数据集:
     Video-R1-CoT-165K: 用于 SFT 冷启动（含 chain-of-thought 标注）
     Video-R1-260K: 用于 RL 训练（图像+视频混合）

性能:
  Video-R1-7B 在视频空间推理基准 VSI-Bench 上达到 37.1%
  → 超越 GPT-4o（闭源商业模型）
  在 VideoMMMU、MVBench、TempCompass 等通用基准上也有显著提升
```

### 4.2 VideoChat-R1 (2025.04, NeurIPS 2025)

**聚焦时空感知能力的强化微调**：

```
核心思路:
  用多任务联合 RL 增强模型的时空感知和推理能力
  任务包括: 时序定位、物体追踪、视频问答等

关键设计:
  1. 时空专用奖励:
     时序定位奖励: 预测的起止时间与 ground truth 的 IoU
     物体追踪奖励: 预测的边界框与实际位置的匹配度

  2. 层级式推理（推理时）:
     模拟人类逐步聚焦注意力的过程
     Step 1: 先粗粒度扫描整个视频
     Step 2: 定位感兴趣区域 (ROI)
     Step 3: 对 ROI 做精细推理
     → 逐步缩小关注范围，提高回答精度

性能提升:
  时序定位: +31.8 (相比基线)
  物体追踪: +31.2
  通用视频问答也有提升，且不损害原有能力
```

### 4.3 视频推理的意义

```
为什么视频推理比图像推理更难:
  图像推理: 所有信息在一张图上，可以反复查看
  视频推理: 信息分散在时间轴上，需要跨帧整合证据
  例: "这个人为什么摔倒？"
    → 需要看到之前地上有水 (帧A) + 人踩到水的位置 (帧B) + 摔倒 (帧C)
    → 需要跨帧因果链推理

视频推理 = 视频理解的"System 2"
  System 1 (快): "视频里有个人在走路" → 直接描述
  System 2 (慢): "这个人在3:20放慢脚步，可能因为2:15看到了告示牌上的警告"
                 → 需要链式推理
```

## 五、视频指令数据构建

### 4.1 数据来源

```
类型 1: 从已有视频数据集转化
  ActivityNet Captions → 视频描述 QA
  MSRVTT → 视频检索 QA
  NExT-QA → 因果/时序推理 QA
  方法: 将原始标注用 GPT-3.5/4 改写为对话格式

类型 2: 用 LLM 生成
  给 GPT-4 提供视频的多帧 caption + 时间戳
  让 GPT-4 生成: 描述、QA、时序推理、因果推理等
  代表: Video-ChatGPT 的 100K 数据

类型 3: 人工标注
  成本高但质量最好
  通常用于高质量评测集，而非训练集
```

### 4.2 视频指令数据的关键维度

```
维度 1: 描述能力
  "描述这个视频的内容"
  "视频中发生了什么？"

维度 2: 时序理解
  "视频中事件的发生顺序是什么？"
  "这个人先做了什么，后做了什么？"

维度 3: 因果推理
  "为什么这个人突然停下来了？"
  "接下来可能会发生什么？"

维度 4: 计数和定位
  "视频中出现了几个人？"
  "红色汽车在第几秒出现的？"

维度 5: 多轮对话
  连续追问视频细节
  需要模型记住前面讨论的内容
```

```
  开源与闭源差距显著缩小
```

### 8.2 2025-2026 关键方向进展

```
方向 1: Video Reasoning（视频推理）  ← 已落地
  Video-R1 (NeurIPS 2025): T-GRPO 算法，7B 模型超越 GPT-4o
  VideoChat-R1 (NeurIPS 2025): 时空感知 RL，时序定位 +31.8
  → 从"看到什么说什么"进化为"看到后链式推理"

方向 2: 视频 Agent  ← 活跃发展中
  视频 VLM 作为 Agent 的感知模块
  结合工具调用: OCR、ASR、搜索引擎
  应用: 视频自动标注、视频内容审核、会议纪要生成
  InternVL3 已支持 Tool-Augmented 视频理解

方向 3: 实时流式视频理解  ← 已实现
  StreamingVLM: 固定 KV Cache，单 H100 实现 8 FPS
  可处理无限长度视频流（2小时+评测）
  应用: 实时视频监控、直播内容理解、AR/VR 交互

方向 4: 音视频联合理解  ← 开源突破
  Qwen3.5-Omni (2026.03): 首个开源全模态模型
  原生 Audio Transformer，113 种语言语音识别
  从"只看画面"到"看+听+说"的完整闭环
  → Gemini 不再是音视频联合的唯一选择

方向 5: 帧采样策略优化  ← 新兴热点
  Frame-Voyager: 查询驱动的智能帧选择
  InfoShot: 镜头感知采样
  → 从"人为设规则"到"模型学习最优采样"

方向 6: 推理效率优化  ← 工程重点
  InternVL3.5 ViCO: 动态压缩，4x 推理加速
  InternVL3.5 DvD: 解耦视觉-语言部署
  Qwen3-VL MoE: 延迟-质量灵活权衡
  → 让视频 VLM 从实验室走向生产环境
```

### 8.3 核心 Takeaway

1. **简单架构赢了**：复杂的时序模块（Q-Former 等）最终不如"直接拼接帧 token + 让 LLM 学"简单有效
2. **Token 压缩是关键瓶颈**：视频理解的核心工程挑战不是时序建模本身，而是如何在有限上下文中塞入足够多的视频信息。动态压缩（ViCO/ViR）是最新解法
3. **图像能力是视频的基础**：所有强视频 VLM 都建立在强图像 VLM 之上，先学好空间理解再扩展到时序
4. **长视频和流式处理已解决**：Qwen2.5-VL/Qwen3-VL 支持小时级视频，StreamingVLM 实现无限流式理解
5. **统一是趋势，全模态是终局**：从图像+视频统一 → 图像+视频+音频+语音统一（Qwen3.5-Omni）
6. **数据质量决定上限**：架构趋同后，高质量视频指令数据成为核心竞争力
7. **从理解到推理**：Video-R1 证明 RL 可以有效提升视频推理，reasoning 能力已从前沿变为标配
8. **帧采样策略被低估**：没有显式时序模块时，采样策略直接决定性能上限，智能采样是下一个优化点
9. **开源追赶加速**：Qwen3-VL、InternVL3.5、Qwen3.5-Omni 在多项基准上已接近或超越闭源模型
