# MoE 详解 (Mixture of Experts)

混合专家模型 (Mixture of Experts, MoE) 是一种通过**稀疏激活**实现"参数量大但计算量小"的模型架构。它是当前大语言模型（如 GPT-4、DeepSeek-V3、Mixtral）背后的核心技术之一。

本文从零开始讲解 MoE 的核心思想、架构设计、工程挑战和前沿进展。

---

## 一、核心思想：为什么需要 MoE？

### 1.1 Dense 模型的困境

传统的 Dense（稠密）模型有一个简单但昂贵的特性：**每个输入 token 都要经过模型的所有参数**。

比如一个 7B 参数的模型，处理每个 token 时，7B 参数全部参与计算。如果你想让模型"更聪明"——存储更多知识、理解更复杂的任务——唯一的办法就是加大参数量。但参数量翻倍，计算量也翻倍，训练和推理的成本都会急剧上升。

这就产生了一个核心矛盾：

> **模型越大越聪明，但越大越贵。能不能"参数很多，但每次只用一部分"？**

### 1.2 MoE 的核心直觉

MoE 的解决思路非常直觉化，可以用一个类比来理解：

> **Dense 模型像一个全科医生**：不管你得了什么病，都是同一个医生看。他什么都会一点，但每个领域都不够精。想让他更专业？只能让他学更多年（增加参数），但看诊时间也更长（计算量更大）。
>
> **MoE 模型像一个医院**：有很多专科医生（专家），前台（路由器）根据你的症状把你分配给最合适的 1-2 个医生。医院有很多医生（参数量大），但你每次只看 1-2 个（计算量小）。

用数字来感受：

```
Dense Model (7B 参数):
  每个 token → 经过全部 7B 参数 → 计算量 ∝ 7B

MoE Model (如 Mixtral 8x7B):
  总参数 ≈ 47B（8 个专家 × 约 7B，加上共享部分）
  每个 token → 路由器选 2 个专家 → 激活约 13B 参数
  计算量 ∝ 13B（接近一个 13B Dense 模型的成本）
  但模型容量 ≈ 47B Dense 模型的水平
```

**核心价值：用 13B 的计算成本，获得接近 47B 模型的能力。**

### 1.3 稀疏激活 vs 稠密激活

这里引出一个关键概念——**稀疏激活 (Sparse Activation)**：

- **稠密激活 (Dense)**：所有参数对所有输入都参与计算。参数利用率 100%，但计算量大。
- **稀疏激活 (Sparse)**：每个输入只激活一部分参数。参数利用率低（如 25%），但计算量小。

MoE 是稀疏激活的一种实现方式：通过路由机制，让每个 token 只"看到"少数几个专家，从而实现稀疏性。

## 二、架构详解

### 2.1 MoE 在 Transformer 中的位置

回忆一下标准 Transformer Block 的结构：

```
输入 x
  │
  ├→ Multi-Head Attention → 残差连接 + LayerNorm
  │
  └→ FFN (Feed-Forward Network) → 残差连接 + LayerNorm
  │
输出
```

**MoE 替换的是其中的 FFN 层**，注意力层完全不变。也就是说：

```
标准 Transformer Block:          MoE Transformer Block:
  Attention                        Attention（不变）
  FFN (单个)                       MoE Layer (多个专家 FFN + 路由器)
```

为什么替换 FFN 而不是 Attention？因为在标准 Transformer 中，**FFN 层占据了大部分参数**（通常约 2/3）。FFN 的作用被认为是"知识存储"——它将注意力层提取的特征映射到更高维空间进行非线性变换。把 FFN 替换成多个专家，相当于大幅增加了模型的"知识存储容量"。

![MoE 架构: 标准 Transformer 层 vs MoE Transformer 层 (Mixtral, Jiang et al., 2024)](assets/moe_architecture.png)

> 图源: *Mixtral of Experts*, Figure 1. 左侧为标准 Transformer Block (单个 FFN)，右侧为 MoE 层 (Router 选择 Top-K 专家并行计算后加权求和)。

### 2.2 MoE 层的三个组件

一个 MoE 层由三部分组成：

#### ① 专家网络 (Expert Networks)

每个"专家"就是一个独立的 FFN，结构和标准 Transformer 的 FFN 完全一样：

$$\text{FFN}(x) = W_2 \cdot \sigma(W_1 \cdot x + b_1) + b_2$$

其中 $\sigma$ 是激活函数（如 SiLU/GELU）。所有专家的网络结构相同，但**参数不同**——它们通过训练学会了处理不同类型的输入。

> **类比**：每个专家就像一个拥有相同学历（网络结构）的医生，但各自的临床经验（参数/权重）不同，因此擅长不同的领域。

#### ② 路由器 / 门控网络 (Router / Gating Network)

路由器决定"这个 token 应该交给哪些专家处理"。它是整个 MoE 架构的核心调度机制。

最基本的路由器就是一个**线性层 + Softmax + TopK**：

**第一步：计算每个专家的得分**

$$\text{logits} = W_g \cdot x$$

其中 $W_g \in \mathbb{R}^{N \times d}$ 是门控权重矩阵，$N$ 是专家数量，$d$ 是隐状态维度，$x \in \mathbb{R}^{d}$ 是当前 token 的隐状态向量。

输出是一个 $N$ 维向量，第 $i$ 个分量表示"这个 token 和第 $i$ 个专家的匹配程度"。

**第二步：Softmax 归一化**

$$\text{scores} = \text{Softmax}(\text{logits})$$

把 logits 转化为概率分布，所有专家的得分之和为 1。

**第三步：TopK 选择**

$$g(x) = \text{TopK}(\text{scores}, K)$$

只保留得分最高的 $K$ 个专家（通常 $K=1$ 或 $K=2$），其余专家的得分置为 0。

**举个具体例子**（假设 8 个专家，TopK=2）：

```
Token "猫" 的隐状态 x 经过路由器：

Softmax 得分: [0.35, 0.05, 0.03, 0.25, 0.02, 0.08, 0.12, 0.10]
                E0    E1    E2    E3    E4    E5    E6    E7

TopK=2 选择:   [0.35, 0,    0,    0.25, 0,    0,    0,    0   ]
                E0 ✓                E3 ✓

归一化权重:     E0: 0.35/(0.35+0.25) = 0.583
                E3: 0.25/(0.35+0.25) = 0.417
```

注意最后一步的**归一化**：被选中的专家的权重会重新归一化，使它们的和为 1。这样保证输出的尺度（scale）不会因为只选了部分专家而变小。

#### ③ 加权求和

$$y = \sum_{i \in \text{TopK}} g_i(x) \cdot E_i(x)$$

每个被选中的专家独立处理输入 $x$，得到各自的输出，然后用路由器给出的权重进行加权求和。

**继续上面的例子**：

```
E0(x) = 专家0处理 "猫" 的隐状态，得到向量 v0
E3(x) = 专家3处理 "猫" 的隐状态，得到向量 v3

最终输出 y = 0.583 × v0 + 0.417 × v3
```

### 2.3 完整的前向传播流程

把上面的三个组件串起来，一个 MoE Transformer Block 的完整前向传播如下：

```
输入 token 的隐状态 x (维度 d)
  │
  ├─→ Multi-Head Attention → 残差连接 + LayerNorm → x'
  │
  ├─→ Router: W_g · x' → Softmax → TopK → 得到选中的专家及权重
  │
  ├─→ 被选中的专家各自独立计算:
  │     Expert_i(x') → 输出 v_i
  │
  ├─→ 加权求和: y = Σ g_i · v_i
  │
  └─→ 残差连接 + LayerNorm → 输出
```

### 2.4 参数量 vs 计算量的数学关系

以 Mixtral 8x7B 为例来具体计算：

**总参数量**：
- Attention 层参数（所有层共享，无专家拆分）：约 ~3B（取决于层数和维度等具体配置）
- 每个专家的 FFN 参数：约 ~5.5B
- 8 个专家 = 8 × 5.5B = ~44B
- 路由器参数：极少（每层一个 $N \times d$ 的矩阵），可忽略
- 总计 ≈ 47B

**每个 token 的激活参数**（TopK=2）：
- Attention 层：~3B（全部激活）
- 2 个专家的 FFN：2 × 5.5B = ~11B
- 总激活 ≈ 13B（约等于一个 13B Dense 模型的计算量）

**效率比**：47B 参数 / 13B 激活 ≈ 3.6 倍。也就是说，你用 13B 的计算成本，获得了 47B 参数量级的模型容量。

## 三、关键问题与解决方案

MoE 的想法虽然优雅，但在实践中面临诸多工程和算法挑战。

### 3.1 负载均衡 (Load Balancing)

#### 问题：专家坍缩 (Expert Collapse)

这是 MoE 训练中**最核心的问题**。

训练初期，某些专家可能偶然在某些 token 上表现稍好，路由器就会倾向于给它们更多 token。更多的 token 意味着更多的梯度更新，这些专家就变得更强，路由器就更倾向于选它们——形成**正反馈循环 (Rich-get-richer)**。

最终可能出现：

```
理想情况:                        专家坍缩:
E0: ████████ 12.5%              E0: ████████████████ 40%
E1: ████████ 12.5%              E1: ██████████████ 35%
E2: ████████ 12.5%              E2: █████ 10%
E3: ████████ 12.5%              E3: ███ 8%
E4: ████████ 12.5%              E4: █ 3%
E5: ████████ 12.5%              E5: █ 2%
E6: ████████ 12.5%              E6: █ 1%
E7: ████████ 12.5%              E7: █ 1%
```

坍缩后，模型实际上退化为只使用 2-3 个专家，浪费了大量参数容量。

#### 解决方案一：辅助负载均衡损失 (Auxiliary Load Balancing Loss)

这是 Switch Transformer (2021) 和 Mixtral (2024) 等模型使用的经典方法。

在训练损失中加入一项额外的惩罚项，鼓励路由器均匀分配 token：

$$L_{total} = L_{task} + \alpha \cdot L_{balance}$$

其中 $L_{task}$ 是标准的语言模型损失（如交叉熵），$L_{balance}$ 是辅助平衡损失：

$$L_{balance} = N \cdot \sum_{i=1}^{N} f_i \cdot p_i$$

各变量含义：
- $N$：专家总数
- $f_i$：在当前 batch 中，实际分配给专家 $i$ 的 token 比例（离散统计量）
- $p_i$：路由器给专家 $i$ 的平均概率（即 batch 中所有 token 在 TopK 之前的 Softmax 输出对专家 $i$ 取平均：$p_i = \frac{1}{T}\sum_{t=1}^{T}\text{Softmax}(W_g \cdot x_t)_i$，连续值，可以反向传播）
- $\alpha$：平衡系数，通常取 0.01 左右

**为什么是 $f_i \cdot p_i$ 而不是单独约束 $f_i$？**

因为 $f_i$ 是 TopK 选择后的离散值（某个 token 要么选了这个专家要么没选），不可导，无法直接反向传播。而 $p_i$ 是 Softmax 输出的连续值，可以求梯度。将两者相乘：
- 如果专家 $i$ 已经被分配了很多 token（$f_i$ 大），$L_{balance}$ 会鼓励降低 $p_i$
- 如果路由器给专家 $i$ 的概率已经很高（$p_i$ 大），$L_{balance}$ 也会受到影响

当所有专家被均匀分配时（$f_i = p_i = \frac{1}{N}$），$L_{balance}$ 取最小值 $\frac{1}{N} \times N = 1$。

**$\alpha$ 的取值很敏感**：
- 太小（如 0.001）：平衡效果不明显，仍会坍缩
- 太大（如 0.1）：过度约束路由器，强制均匀分配，降低了模型性能（因为有些 token 可能确实应该由同一个专家处理）
- 通常取 0.01~0.02

#### 解决方案二：DeepSeek-V3 的无辅助损失均衡

DeepSeek-V3 提出了一种更优雅的方案，**不使用辅助损失**，而是给每个专家添加一个可学习的 bias（偏置项）：

$$\text{scores}_i = \text{Softmax}(W_g \cdot x)_i + b_i$$

- $b_i$ 是专家 $i$ 的偏置，**不通过梯度下降更新**，而是用一个简单的启发式规则在每个训练 step 后调整：

**更新规则**：统计当前 step 中每个专家实际接收的 token 数，与理想均匀值 $T/N$ 对比：

$$b_i \leftarrow b_i + \gamma \cdot \text{sign}\left(\frac{T}{N} - c_i\right)$$

其中 $c_i$ 是专家 $i$ 实际收到的 token 数，$\gamma$ 是一个很小的步长（如 0.001）。

直觉很简单：
- $c_i < T/N$（专家 $i$ 吃不饱）→ $b_i$ 增大 → 下次更容易被选中
- $c_i > T/N$（专家 $i$ 太忙了）→ $b_i$ 减小 → 下次少分一点给它

**关键：选择与加权的解耦**

$b_i$ 只用于 TopK 的**选择阶段**，不参与后续的**加权求和**。整个过程分为两步：

1. **选择阶段（用 bias）**：在 Softmax 得分上加 $b_i$，基于加 bias 后的得分做 TopK 选择——此时得分不再求和为 1，但没关系，只需要比较大小来决定选哪几个专家。
2. **加权阶段（不用 bias）**：选完专家后，加权求和使用的是**原始 Softmax 得分**（不含 $b_i$），并对被选中专家的原始得分重新归一化，使权重和为 1。

```
原始 Softmax 得分:  [0.35, 0.05, 0.03, 0.25, ...]
加 bias 后的得分:    [0.35, 0.05, 0.03, 0.28, ...]  ← 仅用于 TopK 选择
                                          ↑ 假设 b₃ = 0.03

TopK=2 选中: E0, E3（基于加了 bias 的得分）

加权求和用原始得分归一化:
  E0: 0.35/(0.35+0.25) = 0.583
  E3: 0.25/(0.35+0.25) = 0.417  ← 用的是原始的 0.25，不是 0.28
```

这种解耦设计保证了 bias 不进入前向计算图、不产生梯度，因此不会干扰主任务（语言模型）的梯度流。

**优势**：
- 辅助损失会"污染"主任务的梯度（$\alpha \cdot L_{balance}$ 的梯度会干扰语言模型的学习），而 bias 调整是解耦的
- 不需要调节敏感的 $\alpha$ 超参数
- 实验表明在大规模训练中效果优于辅助损失方法

### 3.2 Expert Capacity（专家容量）

#### 问题

在实际的 GPU 计算中，我们需要对 tensor 做 **batch 化处理**——所有专家最好同时开始、同时结束计算，才能高效利用硬件。但如果某个专家被分配了特别多的 token，其他专家早已算完，还要等它，造成 GPU 空闲浪费。

#### 解决方案：容量因子 (Capacity Factor)

给每个专家设定一个最大处理 token 数：

$$\text{Expert Capacity} = \text{CF} \times \frac{T}{N}$$

其中：
- $T$：当前 batch 中的 token 总数
- $N$：专家数
- $\frac{T}{N}$：理想情况下每个专家应处理的 token 数
- $\text{CF}$：容量因子（Capacity Factor），通常 1.0~1.5

**CF 的取值影响**：

```
CF = 1.0: 每个专家最多处理 T/N 个 token（完美均匀）
          → 负载不均时会有很多 token 被丢弃
CF = 1.5: 每个专家最多处理 1.5 × T/N 个 token（留有余量）
          → 丢弃更少，但可能浪费计算资源（有些专家没填满）
CF = 2.0: 很少丢弃，但内存和计算开销大
```

**Token Dropping（token 丢弃）**：超出容量的 token 不会被任何专家处理，直接通过残差连接传递到下一层。这在训练时是可接受的（偶尔丢几个 token 影响不大），但在**推理时不能丢弃**——每个 token 都很重要。因此推理时通常不设容量限制，或者用更大的 CF。

### 3.3 通信开销 (Communication Overhead)

#### 问题

在多 GPU 训练/推理中，不同的专家通常分布在不同的 GPU 上（专家并行, Expert Parallelism）。当路由器决定某个 token 要交给某个专家时，这个 token 的数据可能需要从一块 GPU 发送到另一块 GPU。

这种通信模式叫做 **All-to-All 通信**：每块 GPU 都可能需要向其他任意 GPU 发送/接收数据。

```
假设 4 块 GPU，每块各放 2 个专家:
  GPU 0: E0, E1     GPU 1: E2, E3
  GPU 2: E4, E5     GPU 3: E6, E7

GPU 0 上的 4 个 token 路由结果:
  t1 → E0 (本地)     → 不需要通信
  t2 → E3 (GPU 1)   → 需要 GPU0 → GPU1 传输
  t3 → E5 (GPU 2)   → 需要 GPU0 → GPU2 传输
  t4 → E1 (本地)     → 不需要通信

同时，GPU 1/2/3 上的 token 也在做类似的跨 GPU 路由...
```

#### 为什么这是瓶颈？

1. **通信量大**：All-to-All 的通信量与 token 数和专家分布有关，可能占训练时间的 20-40%
2. **不规则通信模式**：每个 token 去哪个专家是动态决定的，无法提前优化通信路径
3. **GPU 间带宽有限**：NVLink 带宽 ~600 GB/s，但跨节点（通过 InfiniBand）可能只有 ~50 GB/s

#### 怎么缓解？

**核心原则：把通信量大的操作放在带宽高的地方。**

All-to-All 通信量最大且不规则，所以要尽量让它发生在**同节点内**（NVLink ~600 GB/s），避免跨节点（InfiniBand ~50 GB/s，差 12 倍）。

具体做法是**组合多种并行策略**，让不同类型的通信待在合适的位置：

```
同一节点内 (NVLink，带宽高):
  ├── TP (张量并行): Attention + 共享专家切分到 4 卡 → AllReduce 通信
  └── EP (专家并行): 路由专家分散在这 4 卡上 → All-to-All 通信

跨节点 (InfiniBand，带宽低):
  ├── PP (流水线并行): 前 30 层在节点 A，后 30 层在节点 B → 只传层间激活值（量小）
  └── DP (数据并行): 不同节点处理不同 batch → 只在更新参数时同步梯度
```

为什么这样能减少跨节点通信？因为 All-to-All（通信量最大）被限制在节点内，跨节点只走 PP 的层间传递和 DP 的梯度同步，这两者的通信量都远小于 All-to-All。

其他优化：
- **通信与计算重叠**：All-to-All 传输时同时计算共享专家（共享专家不需要通信，正好填充等待时间）
- **FP8 量化通信**：DeepSeek-V3 将传输数据从 BF16 压缩为 FP8，通信量减半

> 更详细的并行组合部署方案见 [AI Infra概述](../../大模型/推理优化/AI_Infra概述.md)。

### 3.4 训练不稳定性

MoE 的训练比 Dense 模型更容易出现不稳定。下面逐一分析原因和对应的缓解手段。

#### 问题一：未被选中的专家收不到梯度

TopK 选择是离散的——选了就是选了，没选就是没选。**没被选中的专家根本不参与前向计算**，因此不会产生对 loss 的贡献，梯度链在这里断了：

```
8 个专家，TopK=2，选中 E0 和 E3:

  E0 (被选中): x → FFN_0(x) → 参与 loss 计算 → ✅ 梯度正常回传
  E3 (被选中): x → FFN_3(x) → 参与 loss 计算 → ✅ 梯度正常回传
  E1 (没选中): 没有执行 → 没有产生输出 → ❌ 无梯度，参数不更新
  E2, E4~E7:   同上 → ❌ 全部收不到梯度
```

注意：**被选中的专家的梯度路径是完全可导的**。MoE 的输出 $y = \sum_{i \in \text{TopK}} g_i \cdot E_i(x)$ 中，权重 $g_i$ 来自 Softmax（连续可导），$E_i(x)$ 是 FFN（连续可导），梯度可以通过 $g_i$ 正常传回路由器的 $W_g$。问题不在被选中的专家，而在**没被选中的专家完全没有学习机会**。

如果不加干预，路由器会越来越偏向已经训练好的少数专家（因为只有它们有梯度更新），其他专家永远得不到训练 → 专家坍缩。

**现代做法：不修复梯度，而是保证每个专家都有机会被选中**

早期研究曾用 Straight-Through Estimator (STE) 尝试给未选中的专家近似梯度，但现在的主流做法是**绕开这个问题**——通过其他机制确保所有专家都能轮流获得训练：

| 机制 | 怎么帮助未选中的专家 |
|------|---|
| **Noisy Gating** | 加噪声让不同 step 选中不同专家，轮流训练（见下文问题三） |
| **辅助损失 / bias** | 强制路由分布均匀，每个专家都能分到 token（见 3.1 节） |
| **细粒度小专家** | 256 个专家选 8 个（激活比 3.1%），每个专家被选中的概率远高于 8 选 2（25%） |
| **共享专家** | 通用知识不依赖路由，减少对路由精确性的要求 |

这些机制组合使用后，专家坍缩在实践中已经基本被解决，不再需要对 TopK 本身做梯度近似。

#### 问题二：专家之间竞争（路由器更新的全局影响）

路由器的参数 $W_g$ 是被**所有专家共享**的。$W_g$ 是一个 $N \times d$ 的矩阵，所有 token 共用它来计算路由得分（$\text{logits} = W_g \cdot x$）。当 $W_g$ 更新 $\Delta W_g$ 时，**所有 token 的路由得分都会同时改变**：

$$\text{token}_i \text{ 的新得分} = (W_g + \Delta W_g) \cdot x_i = W_g \cdot x_i + \Delta W_g \cdot x_i$$

$\Delta W_g$ 对每个 token 的影响取决于 $\Delta W_g \cdot x_i$，不同 token 的 $x_i$ 不同，所以影响也不同且无法预测。这意味着：**针对某个 token 优化的 $W_g$ 更新，可能意外地改变其他 token 的路由**。

用一个具体例子来看（假设 4 个专家，TopK=1）：

```
═══ Step 100：路由已经比较稳定 ═══

  "猫吃鱼" (语法类)  → E0  (得分: E0=0.8, E1=0.1, E2=0.05, E3=0.05)
  "1+1=2"  (数学类)  → E1  (得分: E0=0.1, E1=0.7, E2=0.1,  E3=0.1)
  "DNA复制" (生物类) → E2  (得分: E0=0.05, E1=0.05, E2=0.8, E3=0.1)
  "光速"   (物理类)  → E3  (得分: E0=0.05, E1=0.1, E2=0.05, E3=0.8)

各专家已经在各自领域训练了很久，loss 在稳步下降。

═══ Step 101：W_g 更新后——意外的副作用 ═══

这次更新的梯度主要来自 "1+1=2"（希望把数学 token 转给 E3 试试）。
但 W_g 是共享的，更新后所有 token 的得分都变了：

  "猫吃鱼" → E0  (得分: E0=0.6, E1=0.15, E2=0.1, E3=0.15)  ← 还是 E0，但没那么确定了
  "1+1=2"  → E3  (得分: E0=0.1, E1=0.3, E2=0.1, E3=0.5)   ← 如期从 E1 跳到 E3
  "DNA复制" → E3  (得分: E0=0.1, E1=0.1, E2=0.4, E3=0.4)   ← 意外从 E2 跳到 E3！
  "光速"   → E3  (得分: E0=0.05, E1=0.1, E2=0.1, E3=0.75)  ← 还是 E3

结果:
  E0: 收到 "猫吃鱼"（正常）
  E1: 空了（原来的数学 token 被抢走）
  E2: 空了（生物 token 意外跑了）
  E3: 收到 "1+1=2" + "DNA复制" + "光速"（突然爆满，且要处理不擅长的类型）

E3 本来只擅长物理，突然要处理数学和生物 → loss 飙升

═══ Step 102：loss 飙升后 ═══

loss 大 → 梯度大 → W_g 又被大幅更新 → 路由又剧烈变化
→ 可能推回原来的分配，也可能推向另一个完全不同的方向 → 震荡
```

这就像一个公司频繁重组——每个季度把员工调到不同部门，没人能做好自己的工作。Dense 模型不存在这个问题，因为没有路由，每个参数的"职责"是稳定的。

**早期缓解：路由器使用独立的、更小的学习率**

```
主网络 (Attention + 专家 FFN): lr = 3e-4   ← 正常学习
路由器 W_g:                     lr = 3e-5   ← 慢 10 倍
```

小学习率让 $\Delta W_g$ 很小，每步只微调路由得分，**不足以翻转 TopK 排名**：

```
大学习率: Step 100 → Step 101，得分从 E1=0.7 直接跳到 E3=0.5，排名翻转
小学习率: Step 100 → Step 101，得分从 E1=0.70 变为 E1=0.68，排名不变
          → 经过几十步才慢慢切换，专家有时间适应
```

代价是训练更慢。

**现代做法：从架构层面让路由变化的冲击变小**

当前最先进的方案（如 DeepSeek-V3）不靠压学习率，而是从架构设计上降低路由波动的影响：

| 机制 | 为什么能减少震荡 |
|------|---|
| **细粒度小专家**（256 选 8） | 路由翻转时只替换 8 个中的 1-2 个小专家，对输出的冲击只有 ~1/8，而非 2 选 1 时的 ~1/2 |
| **共享专家** | $y = E_{shared}(x) + \sum g_i \cdot E_i(x)$，共享专家始终提供稳定的基线输出，路由专家的波动被"压住" |
| **bias 解耦** | $W_g$ 只收到任务梯度，负载均衡通过 bias 启发式调整（不走梯度），$W_g$ 的梯度更干净、方向更稳定 |

这些机制协同作用后，路由天然更稳定，不再需要刻意压低学习率。

#### 问题三：负载不均的级联效应

负载不均不是一步发生的，而是通过梯度更新在 batch 之间逐步放大：

```
Step 1: E0 偶然多收了一些 token → E0 获得更多梯度更新 → E0 变强
Step 2: 路由器发现 E0 更强 → 给 E0 更多 token → E0 更强
Step 3: 其他专家越来越弱 → 路由器更不选它们 → 它们完全废掉
  ...
最终: 只有 2-3 个专家在工作（专家坍缩，见 3.1 节）
```

这和 3.1 节的专家坍缩是同一个现象的两个视角：3.1 讨论的是"怎么用损失函数/bias 约束路由分布"，这里强调的是"如果不约束，这种不均衡会通过梯度更新自我放大"。

**缓解：Noisy Gating（噪声门控）**

在路由 logits 上加随机噪声，增加探索性：

$$\text{logits}_i = W_g \cdot x + \epsilon_i, \quad \epsilon_i \sim \mathcal{N}(0, \sigma^2)$$

```
无噪声:
  scores: [0.35, 0.33, 0.20, 0.12]
  → 总是选 E0，E1 永远选不上（虽然只差 0.02）

加噪声 (某次):
  scores: [0.35+0.01, 0.33+0.05, 0.20-0.02, 0.12+0.01]
        = [0.36,      0.38,      0.18,      0.13     ]
  → 这次选了 E1！E1 获得了一次训练机会
```

类似强化学习中的 **exploration**——不总是选当前最优的，偶尔探索一下其他选项，防止过早锁死在局部最优。训练后期可以逐渐减小噪声（$\sigma$ 退火）。

#### 问题四：路由 logits 数值爆炸

训练过程中，路由器的 logits 可能越来越大，导致 Softmax 输出越来越极端：

```
训练早期 logits: [1.2, 1.0, 0.8, 0.5]  → Softmax: [0.31, 0.26, 0.21, 0.16] 比较均匀
训练后期 logits: [12,  10,   8,   5 ]  → Softmax: [0.85, 0.12, 0.02, 0.01] 极端集中
```

logits 越大 → Softmax 越"硬" → 梯度要么很大要么接近 0（Softmax 饱和区） → 数值不稳定，可能出现 NaN。

**缓解：z-loss**

约束路由器 logits 的绝对值不要太大：

$$L_z = \frac{1}{T} \sum_{t=1}^{T} \left(\log \sum_{i=1}^{N} e^{z_{t,i}}\right)^2$$

其中 $z_{t,i}$ 是 token $t$ 对专家 $i$ 的原始 logit（Softmax 之前的值）。

$\log\sum e^{z_i}$ 在 logits 绝对值增大时会增大，对其平方做惩罚就是在鼓励 logits 保持在合理范围内。直觉上，z-loss 就像给路由器加了一个"别太自信"的约束，防止它过度确定地把 token 分给某个专家。

#### 缓解手段总结

| 问题 | 缓解手段 | 原理 |
|------|---------|------|
| 未选中专家无梯度 | Noisy Gating + 辅助损失/bias + 细粒度专家 | 保证每个专家都有机会被选中并训练 |
| 专家竞争 / 路由震荡 | 路由器独立小学习率 | 路由器慢慢学，减少全局扰动 |
| 负载不均级联放大 | Noisy Gating + 辅助损失/bias | 增加探索性 + 约束路由分布 |
| Logits 数值爆炸 | z-loss | 惩罚 logits 过大，保持 Softmax 平滑 |
| 整体不稳定 | 较小的全局学习率 | 暴力但有效，代价是训练更慢 |

### 3.5 训练推理不一致性

MoE 的训练过程中为了效率做了许多妥协（token dropping、容量限制、辅助损失/bias 干预、大 batch 路由），这些在推理时都不存在或行为不同，导致训练和推理之间存在固有的 gap。

#### 不一致的来源

| 训练时 | 推理时 | 不一致 |
|--------|--------|--------|
| Token Dropping（超出容量的 token 被丢弃） | 不能丢弃任何 token | 训练时模型从未学过"所有 token 都被处理"的情况 |
| Capacity Factor 限制每个专家的 token 数 | 通常不设上限 | 推理时的路由不受容量约束 |
| 辅助损失人为干预路由分布 | 无辅助损失 | 路由器的学习目标在训练和推理间不同 |
| 大 batch（数千 token 一起路由） | 逐 token 自回归生成 | 路由统计特性不同 |

#### Token Dropping 是核心矛盾

训练时因为 Capacity Factor 限制，超出容量的 token 通过残差连接直接跳过 MoE 层。推理时不 drop、每个 token 都被完整处理。

这看起来推理时"更正确"，但问题在于：**模型在训练时见到的是"偶尔有 token 跳过某层"的情况，推理时却要求每个 token 都被处理**。可以类比 Dropout——训练时随机丢弃神经元，推理时全部保留。

**为什么实际影响不大**：

1. **残差连接兜底**：被 drop 的 token 不是完全丢失，`x → 0 + x → 下一层`，只是在这一层少了一次专家处理，其他几十层仍然正常
2. **Drop 率很低**：合理设置 CF（如 1.25）后，被 drop 的 token 比例通常 < 1%，相当于轻微的噪声
3. **隐式正则化**：和 Dropout 类似，训练时偶尔跳过处理反而可能起到正则化效果

#### 辅助损失的不一致

使用辅助损失时，训练的目标函数是 $L_{task} + \alpha \cdot L_{balance}$，推理时只有 $L_{task}$ 的行为。辅助损失的梯度在训练时会影响路由器 $W_g$ 的学习方向——路由器不仅在学"怎么路由最好"，还在学"怎么路由最均匀"，这两个目标有时是矛盾的。

DeepSeek-V3 的 bias 方案缓解了这个问题：bias 不进入梯度，推理时保留最终 bias 值，训练推理行为一致。

#### 解决方案

| 方案 | 思路 | 代表 |
|------|------|------|
| **Drop-less MoE** | 训练时也不丢弃 token，通过动态 tensor 重组让每个专家处理不定长序列。属于过渡方案（2022-2023），现已被细粒度专家方案取代 | Tutel (2023), MegaBlocks (2022) |
| **细粒度小专家** | 256 个小专家让负载天然更均匀，drop 率趋近于零 | DeepSeek-V3 |
| **无辅助损失均衡** | 用 bias 替代辅助损失，训练推理使用同一套路由逻辑 | DeepSeek-V3 |
| **共享专家兜底** | 即使路由专家分配在推理时有偏移，共享专家始终处理所有 token，降低对路由精确性的依赖 | DeepSeek-V2/V3 |
| **蒸馏为 Dense** | 彻底消除 MoE 的路由机制，训练推理天然一致 | DeepSeek-R1 蒸馏版 |

当前的趋势是**从架构层面缩小训练推理 gap**（细粒度专家 + 共享专家 + 无辅助损失 + drop-less），而非事后弥补。实在对一致性要求极高的场景，蒸馏为 Dense 是终极方案。

## 四、路由策略的演进

路由器的设计是 MoE 研究的核心方向之一，随着研究深入，出现了多种路由策略。

### 4.1 Token Choice（Token 选择专家）⭐ 当前主流

这是目前所有主流 MoE 大模型采用的路由策略：

- **每个 token 选择 TopK 个专家**
- 优点：每个 token 的计算量固定（恰好 K 个专家），且天然适配自回归解码（逐 token 路由即可）
- 缺点：可能导致某些专家过载、某些专家闲置（需要负载均衡机制配合）
- 代表：Switch Transformer (TopK=1)、Mixtral (TopK=2)、DeepSeek-V2/V3、Qwen-MoE

Token Choice 的负载均衡问题在早期是主要痛点，但随着细粒度小专家、无辅助损失 bias 等技术的成熟（见 3.1 节），这一缺点已被大幅缓解。

### 4.2 Expert Choice（专家选择 Token）📖 学术探索，未被主流采用

由 Zhou et al. (2022) 提出的反向思路：

- **每个专家从所有 token 中选择自己要处理的 TopK 个 token**
- 优点：每个专家的负载天然均衡（每个专家处理固定数量的 token）
- 缺点：某些 token 可能被很多专家选中（计算浪费），某些 token 可能不被任何专家选中（信息丢失）
- **致命缺点：不适用于自回归解码**——生成 token 时是逐个产生的，没有"所有 token"让专家挑选。训练时可以用（有完整序列），推理时用不了，又回到了训练推理不一致的问题（见 3.5 节）

因此目前没有主流大模型采用此方案。它的核心贡献是证明了"反向选择可以天然均衡负载"这一思路，但实际落地的都是 Token Choice + 负载均衡机制的组合。

### 4.3 TopK 的选择

| TopK | 特点 | 代表模型 |
|------|------|---------|
| K=1 | 最稀疏，计算量最小；但可能不够灵活 | Switch Transformer |
| K=2 | 最常用的平衡点；两个专家的组合提供了足够的表达能力 | Mixtral, GPT-4 (传闻) |
| K=4~8 | 更大的 K 配合更多更小的专家 | DeepSeek-V2 (K=6), DeepSeek-V3 (K=8) |

**K 越大**，每个 token 激活的参数越多，计算量也越大。但如果同时增加专家数量（比如 DeepSeek-V3 用 256 个小专家选 8 个），仍然可以保持低激活比。

## 五、主流 MoE 模型对比

| 模型 | 总参数 | 激活参数 | 专家数 | TopK | 共享专家 | 路由策略 | 特点 |
|------|--------|---------|--------|------|---------|---------|------|
| **Switch Transformer** (2021) | - | - | 128 | 1 | 无 | Token Choice | 首个大规模 MoE；TopK=1 的简洁设计 |
| **Mixtral 8x7B** (2024) | 47B | ~13B | 8 | 2 | 无 | Token Choice | 首个开源高质量 MoE LLM |
| **Mixtral 8x22B** (2024) | 176B | ~44B | 8 | 2 | 无 | Token Choice | Mixtral 的更大版本 |
| **DeepSeek-V2** (2024) | 236B | 21B | 160 | 6 | 2 | Token Choice | 细粒度专家 + 共享专家 + MLA |
| **DeepSeek-V3** (2024) | 671B | 37B | 256 | 8 | 1 | Token Choice | 无辅助损失均衡、FP8 训练 |
| **Qwen1.5-MoE** (2024) | 14.3B | 2.7B | 60 | 4 | 4 | Token Choice | 小型高效 MoE |

### 关键观察

1. **专家数量的趋势**：从 8 个大专家 → 256 个小专家。更多更小的专家提供更细粒度的路由。
2. **共享专家越来越普遍**：DeepSeek-V2/V3、Qwen-MoE 都引入了共享专家（见下节）。
3. **激活比 (激活参数/总参数) 在降低**：Mixtral 约 28%，DeepSeek-V3 约 5.5%——稀疏程度越来越高。

## 六、DeepSeek 的 MoE 创新

DeepSeek 系列模型在 MoE 上做了大量前沿创新，代表了 MoE 研究的最新进展。

### 6.1 细粒度专家 (Fine-grained Experts)

**传统 MoE**：8 个大专家，每个专家是一个完整的大 FFN，TopK=2。

**DeepSeek 的做法**：把 1 个大专家拆成多个小专家。比如 DeepSeek-V3 有 256 个小专家，TopK=8。

**为什么这样更好？** 关键在于**组合空间**的差异：

```
传统 8 专家选 2:
  可能的组合数 = C(8,2) = 28 种

DeepSeek 256 专家选 8:
  可能的组合数 = C(256,8) ≈ 4.5 × 10^13 种
```

$C_{256}^{8}$ 远大于 $C_{8}^{2}$，这意味着模型可以为不同的 token 提供**天文数字级的差异化处理路径**。不同类型的输入可以被路由到完全不同的专家组合，实现更精细的特征处理。

**实现细节**：
- 将原来一个大 FFN（中间维度 $d_{ff}$）拆成 $m$ 个小 FFN（中间维度 $d_{ff}/m$）
- 总参数量不变，但路由灵活性大幅提升
- 代价是路由器的 $W_g$ 矩阵变大（从 $8 \times d$ 变成 $256 \times d$），但这部分参数很小

### 6.2 共享专家 (Shared Experts)

**动机**：在纯 MoE 中，所有知识都分散在路由专家中。但有些知识是"通用"的（如语法规则、常见词的语义），几乎每个 token 都需要。如果这些知识被分散到不同专家，每个 token 都要"碰运气"才能被路由到包含这些通用知识的专家。

**解决方案**：设置若干**始终激活**的共享专家，不经过路由器：

```
Token x
  │
  ├─→ Router → TopK 路由专家（稀疏激活）→ 输出 y_routed
  │
  └─→ 共享专家（始终激活）            → 输出 y_shared
  │
  最终输出 = y_routed + y_shared
```

**效果**：
- 共享专家负责捕获**通用知识**（语法、常识、高频模式）
- 路由专家专注学习**专业知识**（特定领域、罕见模式）
- 二者互补，提升了整体模型质量

**DeepSeek-V3 的具体配置**：1 个共享专家 + 256 个路由专家（选 8 个），每个 token 实际激活 9 个专家 FFN。

## 七、MoE 的训练与推理

### 7.1 训练

MoE 的训练通常需要**专家并行 (Expert Parallelism)** 与其他并行策略结合：

```
                    ┌─── GPU 0: E0, E1 ───┐
                    │                      │
数据并行 (DP) ──── ├─── GPU 1: E2, E3 ───┤ ──── 专家并行 (EP)
                    │                      │
                    ├─── GPU 2: E4, E5 ───┤
                    │                      │
                    └─── GPU 3: E6, E7 ───┘
```

训练中的关键考虑：
- **All-to-All 通信**是主要瓶颈，通常占总训练时间的 20-40%
- 需要仔细调节 batch size 和序列长度，以平衡通信和计算
- DeepSeek-V3 使用 FP8 混合精度训练，降低通信数据量和计算成本

### 7.2 推理

MoE 推理有一个重要的特性——**推理显存和 Dense 同参数量模型一样大**：

```
Dense 7B:   需要加载 7B 参数到 GPU 显存
MoE 47B:    需要加载 47B 参数到 GPU 显存（虽然只用 13B）
```

这是因为路由是动态的——你不知道每个 token 会被路由到哪个专家，所以**所有专家的参数都必须常驻显存**。这是 MoE 推理部署的主要挑战。

**推理优化策略**：
- **Expert Offloading**：把不常用的专家参数卸载到 CPU/SSD，需要时再加载（会增加延迟）
- **Expert Parallelism**：多 GPU 分担不同专家，通过高速互联通信
- **量化**：将专家参数量化为 INT4/INT8，减少显存占用
- **专家缓存 (Expert Caching)**：根据历史统计，预测哪些专家即将被使用

## 八、MoE vs Dense 的全面对比

| 维度 | MoE | Dense |
|------|-----|-------|
| **同等计算量下的性能** | 更强（可利用更多参数存储知识） | 较弱 |
| **训练效率 (FLOPS)** | 更高（同 FLOPS 预算下可训练更大模型） | 标准 |
| **训练稳定性** | 更难（路由不稳定、负载不均、梯度问题） | 更稳定 |
| **推理显存** | 更大（需加载全部参数到显存） | 更小（参数量=计算量） |
| **推理延迟 (单 token)** | 可能更高（通信开销、专家调度） | 更可预测 |
| **推理吞吐 (批量)** | 可能更高（稀疏计算利用率好） | 标准 |
| **工程复杂度** | 高（路由、负载均衡、通信、并行策略） | 低 |
| **模型蒸馏** | 可蒸馏为 Dense 模型（如 DeepSeek-R1 蒸馏版） | 不需要 |
| **端侧部署** | 困难（显存大、依赖通信） | 更适合 |
| **最佳场景** | 云端部署、追求极致性能、计算预算充足 | 端侧部署、资源受限、简单场景 |

### 什么时候选 MoE？

- 你有足够的 GPU 显存来放下全部参数
- 你追求在**固定计算预算**下的最佳性能
- 你的应用场景对延迟不是极端敏感（允许通信开销）
- 你有经验处理 MoE 训练中的各种工程问题

### 什么时候选 Dense？

- 端侧 / 移动端部署，显存非常有限
- 需要简单、可预测的推理行为
- 团队没有 MoE 训练/部署的工程经验
- 模型规模不大（< 10B），MoE 的优势不明显

## 九、MoE 的历史脉络

MoE 并不是 LLM 时代的发明，它有很长的历史：

| 时间 | 里程碑 | 贡献 |
|------|--------|------|
| 1991 | Jacobs et al. 提出 MoE | 原始概念：多个专家 + 门控网络 |
| 2017 | Shazeer et al. (Google) | 首次将 MoE 应用于大规模 LSTM 语言模型 (137B 参数) |
| 2021 | Switch Transformer (Google) | 首个大规模 Transformer MoE；TopK=1 的简洁设计 |
| 2022 | ST-MoE (Google) | 系统性研究 MoE 训练稳定性 |
| 2023 | GPT-4 (OpenAI) | 传闻使用 MoE 架构 |
| 2024 | Mixtral 8x7B (Mistral) | 首个高质量开源 MoE LLM |
| 2024 | DeepSeek-V2/V3 | 细粒度专家、共享专家、无辅助损失均衡 |

## 十、总结

### 一句话总结

> MoE 的核心价值：**用更少的计算获取更多的参数容量**，让模型在固定推理预算下存储更多知识。代价是工程复杂度显著增加。

### 关键概念速查

| 概念 | 含义 |
|------|------|
| 稀疏激活 | 每个 token 只激活部分专家，而非全部参数 |
| 路由器 | 决定 token 分配给哪些专家的门控网络 |
| TopK | 每个 token 选择的专家数量 |
| 专家坍缩 | 路由器偏向少数专家，多数专家闲置 |
| 辅助损失 | 在训练损失中加入平衡项，鼓励均匀分配 |
| 共享专家 | 始终激活的专家，捕获通用知识 |
| 细粒度专家 | 将大专家拆分为多个小专家，增大组合空间 |
| Expert Capacity | 每个专家单次处理的 token 上限 |
| All-to-All 通信 | MoE 中 token 跨 GPU 路由的通信模式 |
