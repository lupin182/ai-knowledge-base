# 数据工程

大模型的数据工程是决定模型质量的最关键因素之一。"Garbage in, garbage out" 在大模型时代尤其成立。

### 1.3 去重（Deduplication）—— 面试常考

去重是预训练数据处理中**最重要的步骤之一**。重复数据会导致模型记忆而非泛化，且浪费计算。

#### 精确去重 (Exact Dedup)

```
文档级: 计算整篇文档的 SHA-256 hash，hash 相同 → 完全重复 → 删除
行级:   计算每一行的 hash，跨文档去重相同行（如版权声明、导航栏）
```

#### 模糊去重 (Fuzzy Dedup) — MinHash + LSH

**Step 1: Shingling（构建 n-gram 集合）**

```
文档 A: "The cat sat on the mat"
5-gram shingles: {"The c", "he ca", "e cat", " cat ", "cat s", "at sa", ...}

文档 B: "The cat sat on the rug"
5-gram shingles: {"The c", "he ca", "e cat", " cat ", "cat s", "at sa", ...}

两个文档共享大部分 shingles → Jaccard 相似度高
```

**Step 2: MinHash（压缩集合为固定长度签名）**

```
问题: n-gram 集合可能有几万个元素，直接比较 Jaccard 太慢
MinHash: 把大集合压缩为 k 个数（签名），k 远小于集合大小

算法:
  准备 k 个独立的 hash 函数 h₁, h₂, ..., h_k
  对集合 S 中的每个元素 x:
    MinHash_i(S) = min_{x∈S} h_i(x)    ← 第 i 个 hash 函数的最小值

  签名: [MinHash_1(S), MinHash_2(S), ..., MinHash_k(S)]

关键定理:
  P(MinHash_i(A) = MinHash_i(B)) = Jaccard(A, B)
  即: 两个集合 MinHash 值相同的概率 = 它们的 Jaccard 相似度

  用 k 个 hash 函数估计: Ĵ(A,B) = (k 个中相等的个数) / k

示例 (k=4):
  文档 A 签名: [12, 45, 78, 23]
  文档 B 签名: [12, 45, 78, 91]  → 3/4 相等 → 估计 Jaccard ≈ 0.75
  文档 C 签名: [56, 89, 34, 67]  → 0/4 相等 → 估计 Jaccard ≈ 0.0
```

**Step 3: LSH（快速找到候选对）**

```
问题: 即使有了签名，N 篇文档两两比较仍需 O(N²)
LSH: 只比较"可能相似"的文档对

做法: 把 k 维签名分成 b 个 band，每个 band 有 r 行 (k = b × r)

  签名 [12, 45, 78, 23, 56, 89]  (k=6)
  分成 b=3 bands, r=2 rows:
    Band 1: [12, 45]
    Band 2: [78, 23]
    Band 3: [56, 89]

  对每个 band: 如果两个文档这个 band 完全相同 → 放入同一个桶 (候选对)

  只要任何一个 band 匹配 → 成为候选对 → 再做精确 Jaccard 检查

概率分析:
  真实 Jaccard = s 的两个文档:
    单个 band 完全匹配的概率: s^r
    至少一个 band 匹配的概率: 1 - (1 - s^r)^b

  s=0.8, b=20, r=5: 匹配概率 ≈ 0.9996  ← 几乎一定找到
  s=0.3, b=20, r=5: 匹配概率 ≈ 0.047   ← 几乎不会误配

  → b 和 r 控制了 "阈值曲线的陡度"
```

**Step 4: 候选对验证与去重决策**

```
对 LSH 找到的候选对:
  1. 计算精确 Jaccard (或用全部 MinHash 签名估计)
  2. 超过阈值 (如 0.8) → 标记为重复
  3. 构建连通分量（A≈B, B≈C → {A,B,C} 是一组重复）
  4. 每组保留一篇，删除其余
```

**实际工程参数（参考 RefinedWeb/Dolma）**：

| 参数 | 典型值 |
|---|---|
| n-gram 大小 | 5 |
| MinHash 签名数 k | 128~256 |
| LSH bands b | 20 |
| rows per band r | 5 |
| Jaccard 阈值 | 0.7~0.8 |

#### 子串去重 (Suffix Array)

用后缀数组找到跨文档的**长公共子串**（如相同的模板文本、抄袭内容），移除重复的子串。

```
做法:
  1. 把所有文档拼接成一个大字符串 (用文档分隔符)
  2. 构建后缀数组 (Suffix Array) + LCP 数组 (最长公共前缀)
  3. LCP > 阈值 (如 100 字符) 的后缀对 → 找到了长重复子串
  4. 移除重复的子串

优势: 能发现文档级去重找不到的"局部重复"
      如: 多篇文章引用同一段法律条文、同一个 Stack Overflow 答案
劣势: 内存消耗大（后缀数组需要原文本 5-10 倍内存）
```

### 1.4 数据配比（Data Mixing）

不同来源的数据按什么比例混合，对模型能力影响巨大：

| 数据类型 | 典型占比 | 影响的能力 |
|---|---|---|
| 网页文本 | 60-70% | 通用语言能力 |
| 代码 | 10-20% | 代码能力 + 逻辑推理（！）|
| 书籍/论文 | 5-10% | 长文本理解、知识深度 |
| 数学 | 3-5% | 数学推理 |
| 百科 | 3-5% | 事实性知识 |
| 多语言 | 5-15% | 多语言能力 |

**关键发现**：
- 代码数据显著提升模型的逻辑推理能力（即使不做代码任务），这是业界共识
- 数学数据对推理能力也有显著帮助
- 多语言数据存在迁移效应：适量的中文数据能提升英文能力（反之亦然）

**主流模型的实际配比（公开信息）**：

| 模型 | 网页 | 代码 | 书籍/论文 | 数学 | 百科 | 其他 |
|---|---|---|---|---|---|---|
| LLaMA-1 | 67% | 4.5% | 4.5% | — | 4.5% | 19.5% (C4+StackExchange) |
| LLaMA-3 | ~50% | ~25% | — | ~10% | — | ~15% (多语言+推理) |
| DeepSeek-V3 | — | ~30% | — | ~20% | — | — (高代码+数学) |

注意 LLaMA-3 大幅增加了代码和数学比例 → 推理能力显著提升。

**数据配比的搜索方法**：

```
方法 1: 基于 Proxy Model
  1. 训练多个小模型 (如 1B)，每个用不同配比
  2. 评估各小模型在目标 benchmark 上的表现
  3. 选最优配比用于大模型训练
  → 假设: 小模型的最优配比 ≈ 大模型的最优配比 (近似成立)

方法 2: DoReMi (Domain Reweighting with Minimax Optimization)
  1. 训练一个 reference model (基础配比)
  2. 训练一个 proxy model (自适应调整配比)
  3. 哪个领域 loss 下降慢 → 增加该领域权重
  → 自动化的配比搜索
```

#### Data Mixing 的深层机制

数据配比不是简单的"各类数据按比例混合"，背后涉及**领域间的正负迁移**：

```
正迁移 (Positive Transfer):
  代码 → 逻辑推理:  代码数据教会模型结构化思考、条件判断、递归分解
  数学 → 推理:      形式化推理能力跨领域迁移
  多语言 → 单语言:  不同语言的共享语义表示互相增强

负迁移 (Negative Transfer):
  低质量网页 → 事实性:  大量噪声数据稀释了高质量知识
  社交媒体 → 正式写作:  口语化表达污染正式输出风格
  领域数据过多 → 通用性: 过度专注某领域导致其他能力退化（"能力税"）

关键认知:
  配比不是线性的——domain A 从 10% → 20% 的收益 ≠ 20% → 30% 的收益
  存在非线性交互: 代码+数学联合增加的效果 > 分别增加之和
```

#### 动态配比（Dynamic Mixing）

```
静态配比的问题:
  训练全程使用固定配比 → 训练前期和后期的最优配比不同
  前期: 模型需要大量通用数据建立语言基础
  后期: 模型需要高质量数据精修特定能力

动态配比策略:
  1. Loss-based Reweighting
     监控各 domain 的 loss 下降速度
     loss 下降慢的 domain → 增加权重（这个 domain 还没学好）
     loss 已经很低的 domain → 降低权重（已经学够了）

  2. Skill-based Mixing
     定义目标能力集合（推理、代码、知识、多语言...）
     定期在 benchmark 上评估各能力
     哪个能力弱 → 增加对应 domain 的权重

  3. 训练阶段切换
     Phase 1 (0-50%): 通用配比，重网页
     Phase 2 (50-80%): 增加代码+数学
     Phase 3 (80-100%): 高质量数据为主（见 1.5 课程学习的退火阶段）
```

#### Data Mixing 与 Scaling 的交互

```
一个重要但常被忽略的问题:
  最优数据配比是否随模型规模变化？

实验发现:
  - 小模型 (1B-7B): 代码数据占比 15-20% 效果最好
  - 大模型 (70B+): 代码数据可以占到 25-30%，因为模型容量够大
  - 原因: 大模型能同时学好多个 domain，不容易出现负迁移

Proxy Model 方法的局限正在于此:
  小模型找到的最优配比 ≠ 大模型的最优配比
  误差随规模差距增大而增大
  → 这就是为什么需要更理论化的框架（如 DoReMi）来替代纯实验搜索
```

### 1.5 课程学习（Curriculum Learning）

#### 基本概念

课程学习的核心思想来自人类学习：**先易后难，循序渐进**。在大模型预训练中，"课程"体现在数据的**呈现顺序**和**质量梯度**。

```
形式化定义:
  给定数据集 D = {d₁, d₂, ..., dₙ}
  课程函数 C(t): 训练步 t → 当前应该使用的数据子集/配比

  与随机 shuffle 的区别:
    随机: 每个 step 均匀随机采样全部数据
    课程: 按 C(t) 有策略地选择数据

  为什么有效:
    1. 避免训练早期被噪声数据干扰（loss landscape 更平滑）
    2. 帮助优化器找到更好的 loss basin
    3. 后期高质量数据的影响更大（靠近最终权重）
```

#### 预训练中的三阶段课程

```
阶段 1 — 广泛学习 (前 80% tokens):
  大量网页+代码数据，学习基础语言模式
  数据质量要求: 中等 (过基本清洗即可)
  学习率: warmup → 保持高位
  目标: 建立通用的语言表示和世界知识

阶段 2 — 质量提升 (后 15% tokens):
  逐步提高高质量数据占比
  增加书籍、论文、高质量教程
  开始混入对话数据
  学习率: 开始缓慢下降
  目标: 精修知识质量，建立更好的内部一致性

阶段 3 — 退火 (最后 5% tokens):
  主要使用高质量数据
  降低学习率 → 线性退火到 0
  这个阶段对最终模型质量影响显著

  LLaMA-3 的退火数据:
  - 高质量网页 (经 classifier 筛选后的 top 5%)
  - 精选代码和数学
  - 教科书风格的合成数据

  退火为什么关键:
  - 最后见到的数据对模型输出分布影响最大（recency bias）
  - 高质量退火数据 = "临考冲刺用精选题"
  - 实验表明退火阶段的数据质量可以让 benchmark 提升 2-5%

类比人类学习: 先广泛阅读 → 再精读经典 → 最后考前冲刺
```

#### 难度度量：怎么定义"容易"和"难"

```
方法 1: 基于 Loss 的难度
  用一个预训练好的小模型给每条数据打分:
    difficulty(d) = loss_small_model(d)
  loss 高的 → 难样本，loss 低的 → 易样本
  训练时先喂 loss 低的，后喂 loss 高的

方法 2: 基于文本特征的难度
  - 文本长度、词汇丰富度、句法复杂度
  - 是否包含推理、数学、代码等复杂内容
  简单粗暴但有效，尤其是长度排序（先短后长）

方法 3: 基于模型自身的难度 (Self-Paced Learning)
  训练过程中动态评估:
    当前模型在每条数据上的 loss → 作为实时难度
    每隔 K 步重新排序数据池
  优点: 难度随模型能力动态调整
  缺点: 计算开销大（需要频繁评估）

实际上大多数团队用的是方法 1 + 阶段式课程:
  不是逐条排序，而是按阶段切换数据质量级别
  因为精细的逐条排序在万亿 token 规模下不现实
```

#### Data Mixing × Curriculum Learning 的联合设计

```
实际训练中，数据配比和课程学习是联合设计的:

LLaMA-3 的完整训练策略（推测）:

  Step 0-2T tokens:    网页 60% / 代码 20% / 数学 5% / 其他 15%
  Step 2T-10T tokens:  网页 50% / 代码 25% / 数学 10% / 其他 15%
  Step 10T-14T tokens: 网页 40% / 代码 25% / 数学 15% / 高质量 20%
  Step 14T-15T (退火): 高质量网页 30% / 精选代码 25% / 精选数学 20% / 教科书 25%

                       ← 数据配比在变 (mixing) →
                       ← 数据质量在升 (curriculum) →

关键: 这两个维度是耦合的，不能独立优化
  - 增加代码比例的时机很重要（太早模型还没学好自然语言）
  - 退火阶段的配比也不同于主训练阶段
```

#### 前沿方向：能否自动学习课程？

```
手动设计课程的问题:
  - 依赖专家经验和大量 ablation 实验
  - 不同规模模型的最优课程不同
  - 调参空间巨大（阶段数 × 每阶段配比 × 切换时机）

自动课程学习 (Automated Curriculum):
  1. Multi-armed Bandit 方法
     把每个 domain 看作一个 arm
     用 bandit 算法动态分配采样概率
     reward = validation loss 的下降速度

  2. Reinforcement Learning for Data Selection
     训练一个"数据选择策略"网络
     输入: 当前模型状态 + 候选数据
     输出: 采样概率
     reward: 模型在目标 benchmark 上的改善

  3. Online Data Mixing (ODM)
     边训练边调整配比，不需要预设阶段
     每隔 N 步评估各 domain 的梯度信号
     → 梯度信号强的 domain 增加权重

这些方法目前主要在小规模实验 (<10B) 中验证
大规模训练中仍以专家设计 + proxy model 验证为主
```

### 1.6 合成数据（Synthetic Data）

高质量数据不够用？用模型自己生成！

```
预训练阶段的合成数据:

1. 教科书风格数据 (phi 系列的核心思路)
   用 GPT-4 生成高质量教科书风格文本:
   "请以教科书的风格解释 [主题]，包含示例和练习"
   → 比原始网页数据质量高得多
   → phi-1.5 用 ~100B token 的合成数据训出了 1.3B 模型，效果媲美 7B 模型

2. 代码数据增强
   - 给代码添加详细注释
   - 生成代码的自然语言解释
   - 从文档生成对应代码

3. 数学推理链
   - 给数学问题生成详细解题步骤 (Chain-of-Thought)
   - 正确解法和错误解法都生成（用于 DPO）

4. 指令回译 (Backtranslation)
   给一段文本 → 用模型生成"什么指令能产生这段文本？"
   → 获得 (instruction, response) 对

风险:
  - Model Collapse: 模型在自己生成的数据上训练 → 分布退化
  - 解决: 合成数据和真实数据混合使用，合成数据占比控制在 30% 以内
```

## 二、SFT 数据

### 2.1 SFT 数据的核心原则

> SFT 不是"教模型新知识"，而是**教模型如何表达它已经有的知识**（对齐格式和风格）。

因此 SFT 数据的**质量远比数量重要**。

| 维度 | 要求 |
|---|---|
| 数据量 | 1K-100K 条（少量精品 > 大量低质）|
| 指令多样性 | 覆盖多种任务类型（QA、总结、翻译、代码、数学...）|
| 回答质量 | 必须是人写或强模型生成的高质量回答 |
| 格式规范 | 统一 Chat Template，正确的 role 标记 |

### 2.2 SFT 数据的构建方式

#### 人工标注

```
流程: 编写标注指南 → 标注员写 (instruction, response) → 交叉审核 → 清洗
优点: 质量最高，可控
缺点: 成本高，速度慢
代表: InstructGPT 的 13K 标注数据
```

#### Self-Instruct / Evol-Instruct

```
Self-Instruct:
  用 LLM 自己生成 instruction → 再用 LLM 生成 response → 过滤低质量

Evol-Instruct (WizardLM):
  对种子指令做进化:
  - 增加约束 ("用 Python 且不用循环")
  - 增加深度 ("解释为什么这个算法是 O(nlogn)")
  - 更具体化 ("给我一个电商场景的推荐系统")
  - 增加步骤 ("分三步完成，每步给出中间结果")
```

#### 从强模型蒸馏

```
用 GPT-4/Claude 生成高质量 response:
  1. 准备多样化的 instruction 集合
  2. 调用强模型 API 生成回答
  3. 人工审核 / 自动评分过滤

代表: Alpaca (52K, GPT-3.5), Vicuna (70K, ShareGPT)
```

### 2.3 SFT 数据常见问题

| 问题 | 现象 | 解决方案 |
|---|---|---|
| 数据不平衡 | 某类任务过多，其他任务能力退化 | 按任务类型采样平衡 |
| 回答过长/过短 | 模型总是输出很长/很短的回答 | 控制训练数据的回答长度分布 |
| 幻觉数据 | 训练数据本身包含错误信息 | 用 ground truth 验证、人工审核 |
| 风格不一致 | 有时正式有时口语 | 统一标注规范 |

## 三、偏好数据（用于 RLHF/DPO）

### 3.1 数据格式

```json
{
  "prompt": "解释量子纠缠",
  "chosen": "量子纠缠是指两个或多个粒子之间存在一种特殊的关联...(详细准确的回答)",
  "rejected": "量子纠缠就是两个东西连在一起...(模糊不准确的回答)"
}
```

### 3.2 偏好数据的构建

```
方法 1: 人工排序
  同一个 prompt，模型生成多个 response → 人工标注哪个更好

方法 2: AI 反馈 (RLAIF)
  用强模型（GPT-4）判断哪个 response 更好
  优点: 成本低、速度快
  缺点: 受限于评判模型的能力和偏好

方法 3: 规则判断
  对于有明确答案的任务（数学、代码），用正确性作为偏好信号
  正确答案 = chosen, 错误答案 = rejected
```

## 四、数据飞轮

成熟的大模型团队会建立**数据飞轮**，持续改进数据质量：

```
┌──────────────┐
│  模型上线     │
└──────┬───────┘
       ▼
┌──────────────┐
│  收集用户反馈 │ ← 点赞/点踩、重新生成、编辑回答
└──────┬───────┘
       ▼
┌──────────────┐
│  识别弱点     │ ← 分析失败 case，找到能力短板
└──────┬───────┘
       ▼
┌──────────────┐
│  针对性补数据 │ ← 为弱点领域标注/生成高质量数据
└──────┬───────┘
       ▼
┌──────────────┐
│  重新训练     │ ← SFT/DPO 迭代
└──────┬───────┘
       ▼
    回到模型上线 → 循环
```

**面试快速回答模板**：

> **Q: 预训练数据怎么清洗？**
> A: URL/语言过滤 → HTML 提取 → 启发式规则过滤（长度/重复/特殊字符）→ 质量分类器 → 去重（MinHash+LSH 模糊去重）→ PII 脱敏。留存率约 5-15%。
>
> **Q: 去重为什么重要？怎么做？**
> A: 重复数据导致模型记忆而非泛化。精确去重用 hash，模糊去重用 MinHash+LSH 估计 Jaccard 相似度，阈值通常 0.8。
>
> **Q: SFT 数据量需要多少？**
> A: 几千到几万条高质量数据即可。SFT 的目标不是注入知识，而是教模型如何正确格式化输出。质量 >> 数量。
