# 预训练 / 后训练团队的日常工作

> 基模（Foundation Model）团队的核心工作拆解。

## 一、预训练团队 (Pre-training Team)

预训练团队的核心任务：**在有限的算力预算下训练出能力最强的基座模型**。

### 日常工作构成

#### 1. 数据工程（占大量时间，最脏最累但最重要）

| 工作 | 具体内容 |
|------|---------|
| **数据采集** | 爬取互联网数据、采购书籍/学术数据、获取代码数据（GitHub）等 |
| **数据清洗** | 去重（MinHash/SimHash）、过滤低质量内容、去除 PII（个人隐私）、去除有害内容 |
| **数据配比实验** | 中英文比例？代码占多少？数学占多少？不同配比对模型能力影响巨大 |
| **数据质量评估** | 用小模型做 ablation，验证新数据源的价值 |
| **数据课程设计 (Curriculum)** | 不同训练阶段喂什么数据？什么时候加入高质量数据？ |
| **合成数据** | 用现有模型生成训练数据（教科书级文本、指令数据、代码等） |

> "基座模型的能力 70% 取决于数据质量"——这是业界普遍共识。

#### 2. 模型架构探索

| 工作 | 具体内容 |
|------|---------|
| **架构改进** | 尝试新的注意力机制（GQA、MLA）、激活函数（SwiGLU）、归一化方式（RMSNorm） |
| **MoE 设计** | 专家数量、路由策略、负载均衡 |
| **长上下文** | RoPE 外推、YaRN、动态 NTK |
| **小规模验证** | 用小模型（1B~7B）做架构消融实验，验证后再 scale up |

#### 3. 训练工程

| 工作 | 具体内容 |
|------|---------|
| **分布式训练** | 配置 TP/PP/DP 并行策略，优化通信效率 |
| **训练稳定性** | 处理 loss spike、NaN、梯度爆炸等问题 |
| **检查点管理** | 定期保存、评估、选择最佳 checkpoint |
| **超参搜索** | 学习率、batch size、warmup 步数等（遵循 µP 等方法论） |
| **故障恢复** | GPU 故障、网络中断后的自动恢复 |
| **集群维护** | 和基础设施团队协作，处理机器问题 |

#### 4. 评估与分析

| 工作 | 具体内容 |
|------|---------|
| **持续评估** | 训练过程中定期在 benchmark 上评估（MMLU、HumanEval、GSM8K 等） |
| **能力分析** | 哪些能力在什么阶段涌现？Loss 下降曲线分析 |
| **对比实验** | 和上一版模型、竞品模型的对比 |
| **Scaling Law 预测** | 用小模型实验预测大模型的最终性能 |

## 二、后训练团队 (Post-training / Alignment Team)

后训练团队的核心任务：**把基座模型变成一个有用、安全、符合人类偏好的产品**。

### 日常工作构成

#### 1. SFT 数据建设

| 工作 | 具体内容 |
|------|---------|
| **数据标注管理** | 制定标注规范、培训标注团队、质量审核 |
| **Prompt 收集** | 从真实用户场景、红队测试、竞品分析中收集多样化的 prompt |
| **回答生成与筛选** | 用模型生成多个回答，人工或自动筛选最优 |
| **数据多样性** | 确保覆盖：问答、写作、代码、数学、多语言、角色扮演等 |
| **蒸馏** | 从更强的模型（如 GPT-4）蒸馏高质量 SFT 数据 |

#### 2. 偏好数据建设

| 工作 | 具体内容 |
|------|---------|
| **偏好标注** | 给定 prompt 和两个回答，人类判断哪个更好 |
| **标注一致性** | 计算标注者之间的一致率 (Inter-annotator agreement) |
| **偏好维度** | 有用性、准确性、安全性、格式合规性等多维度标注 |
| **难样本挖掘** | 专门收集模型容易出错的 case |

#### 3. 训练与实验

| 工作 | 具体内容 |
|------|---------|
| **SFT 训练** | 数据配比实验、学习率调优、过拟合控制 |
| **偏好优化** | DPO / PPO / GRPO 训练，选择技术路线 |
| **RM 训练** | 训练和评估奖励模型的准确度 |
| **消融实验** | 数据量、数据质量、训练方法的 ablation |
| **迭代训练** | 用当前模型生成数据 → 标注 → 训练下一版 |

#### 4. 安全与对齐

| 工作 | 具体内容 |
|------|---------|
| **红队测试** | 尝试诱导模型输出有害内容，发现漏洞 |
| **安全数据构建** | 构建拒绝回答有害请求的训练数据 |
| **过度拒绝分析** | 模型是否在不该拒绝的地方拒绝了？ |
| **合规审查** | 法律、版权、隐私等合规性检查 |

#### 5. 评估与上线

| 工作 | 具体内容 |
|------|---------|
| **自动评估** | Benchmark 分数（MT-Bench、AlpacaEval、Arena-Hard 等） |
| **人工评估** | 内部人员盲测打分，对比新旧版本 |
| **A/B 测试** | 线上小流量测试，收集用户反馈 |
| **Case 分析** | 逐条分析典型 bad case，定位是数据问题还是训练问题 |
| **能力回归测试** | 确保新版本没有在某些能力上退步 |

## 三、两个团队的协作关系

```
预训练团队                              后训练团队
   │                                       │
   │  交付 Base Model                      │
   ├──────────────────────────────────────→ │
   │                                       │
   │  反馈：基座能力不足的方向               │
   │ ←──────────────────────────────────── │
   │  (例如: 数学太弱, 需要加数学数据)       │
   │                                       │
   │  交付改进版 Base Model                 │
   ├──────────────────────────────────────→ │
   │                                       │
   │  RL 训练发现的 insight 反馈            │
   │ ←──────────────────────────────────── │
   │  (例如: DeepSeek-R1 发现 RL 能涌现     │
   │   推理能力, 反过来指导预训练策略)       │
```

## 四、一句话总结

- **预训练团队**：核心是"数据工程 + 训练工程"，目标是把 Scaling Laws 发挥到极致
- **后训练团队**：核心是"数据标注 + 对齐实验"，目标是把基座能力转化为产品体验

> 两个团队的共同点：**大量时间花在数据上，而不是算法上**。算法创新是锦上添花，数据质量是基本功。
