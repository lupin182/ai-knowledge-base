# 推理优化详解

大模型推理的核心矛盾：**模型大、序列长、延迟敏感**。优化围绕 计算、显存、带宽 三个瓶颈展开。

> **KV Cache 相关内容**（显存计算、Prefill vs Decode、PagedAttention、vLLM、Prefix Caching）已拆分为独立文档，详见 **[KV Cache 详解](KV_Cache详解.md)**。

## 一、FlashAttention

### 1.1 标准 Attention 的问题

```
标准 Attention:
Q, K, V ∈ ℝ^{N×d}

1. S = QK^T          → 写入 HBM，大小 N×N    ← 显存 O(N²)
2. P = Softmax(S)    → 读 S，写 P 到 HBM     ← 又一次 O(N²)
3. O = PV            → 读 P，写 O 到 HBM

总共对 HBM 读写 O(N²) 的中间矩阵，3 次！
```

当 N=8192、d=128 时，S 矩阵 = 8192² × 2 bytes = 128 MB **每个头每层**。多头多层加起来显存爆炸。

### 1.2 FlashAttention 的核心思想：Tiling + Online Softmax

**关键洞察**：不需要把 N×N 的注意力矩阵完整写入 HBM。

```
GPU 内存层次:
┌─────────────┐
│   HBM       │  ← 大但慢 (80GB, ~2TB/s)
│  (显存)      │
└──────┬──────┘
       │ 带宽瓶颈
┌──────▼──────┐
│   SRAM      │  ← 小但快 (20MB, ~19TB/s)
│  (片上缓存)  │
└─────────────┘
```

FlashAttention 的做法：

```
将 Q 分成 Br 个 block，K/V 分成 Bc 个 block

for each Q_block:           ← 外循环
    load Q_block 到 SRAM
    初始化 O_block = 0, m = -∞, l = 0   ← 记录当前最大值和指数和

    for each K_block, V_block:   ← 内循环
        load K_block, V_block 到 SRAM
        S_block = Q_block × K_block^T     ← 在 SRAM 里算，不写回 HBM

        # Online Softmax (关键！)
        m_new = max(m, rowmax(S_block))
        P_block = exp(S_block - m_new)
        l_new = l × exp(m - m_new) + rowsum(P_block)
        O_block = O_block × exp(m - m_new) + P_block × V_block
        m = m_new, l = l_new

    O_block = O_block / l       ← 最终归一化
    写 O_block 回 HBM
```

### 1.3 Online Softmax 详解——为什么可以分块算 Softmax？

普通 Softmax 需要**两遍扫描**：第一遍找 max，第二遍算 exp 和 sum。这要求看到整行数据。

**Online Softmax 的推导**（面试可能追问）：

假设已经处理了第 1 个 block，得到了部分结果：
```
m₁ = max(S_block1)                    ← 第 1 块的最大值
l₁ = Σ exp(S_block1 - m₁)             ← 第 1 块的指数和
O₁ = exp(S_block1 - m₁) × V_block1 / l₁   ← 第 1 块的加权输出
```

现在来了第 2 个 block，有新的 max 和 sum：
```
m₂ = max(S_block2)
m_new = max(m₁, m₂)                   ← 全局新最大值

# 关键：之前的 exp 值需要修正！
# 原来: exp(s - m₁)
# 现在: exp(s - m_new) = exp(s - m₁) × exp(m₁ - m_new)
#                       ↑ 原来的值     ↑ 修正系数 (≤1)

l_new = l₁ × exp(m₁ - m_new) + Σ exp(S_block2 - m_new)
        ↑ 修正第1块的sum         ↑ 第2块的贡献

O_new = (O₁ × l₁ × exp(m₁ - m_new) + exp(S_block2 - m_new) × V_block2) / l_new
        ↑ 修正第1块的输出                ↑ 第2块的贡献                      ↑ 重新归一化
```

**本质**：每次来新的 block，用 `exp(m_old - m_new)` 这个衰减因子修正之前的所有累积值。因为 `m_new ≥ m_old`，所以 `exp(m_old - m_new) ≤ 1`，数值稳定。

### 1.4 FlashAttention 的反向传播

训练时需要反向传播计算梯度。标准实现需要保存 N×N 的注意力矩阵 P，FlashAttention 如何处理？

```
前向传播时: 只保存 O (输出), m (行最大值), l (行指数和)
            不保存 S = QK^T 和 P = Softmax(S)

反向传播时: 从 O, m, l 重新计算 P（按 block 重算，不存完整矩阵）
            → 用计算换存储

这叫 "recomputation" / "gradient checkpointing" 的思想
代价: 反向传播多了一次前向计算 → 总计算量约增加 ~25%
收益: 显存从 O(N²) 降到 O(N)，可以处理更长序列
```

### 1.5 复杂度对比

| | 标准 Attention | FlashAttention |
|---|---|---|
| HBM 读写 | $O(N^2)$ | $O(N^2 d / M)$，M 为 SRAM 大小 |
| 额外显存 | $O(N^2)$（存 S 矩阵） | $O(N)$（只存 m, l） |
| Wall-clock | 慢（受限于 HBM 带宽） | 快 2-4x |

### 1.6 FlashAttention-2/3

**FA-1 → FA-2 的关键改进**：

```
FA-1 的并行策略:
  外循环: KV blocks (并行)
  内循环: Q blocks (串行)
  问题: 不同 KV block 写到同一个 Q 的输出 → 需要原子操作，同步开销大

FA-2 的并行策略:
  外循环: Q blocks (并行)       ← 翻转循环！
  内循环: KV blocks (串行)
  优势: 不同 Q block 的输出完全独立 → 无需同步，天然并行

  额外优化:
  - 减少非矩阵乘 FLOPs (rescaling 合并)
  - warp 间更好的工作分配

  结果: MFU 从 ~50% → ~70%，速度提升 ~2x
```

**FA-3（Hopper 架构专属）**：
- 利用 TMA（Tensor Memory Accelerator）异步搬数据：计算和数据搬运同时进行
- 用 WGMMA 指令直接从 shared memory 做矩阵乘，跳过寄存器中转
- 支持 FP8 精度（E4M3），进一步提速
- 结果: H100 上 MFU 达到 ~75%

## 二、模型量化

把模型权重和/或激活值从高精度（FP16/BF16）压缩到低精度（INT8/INT4），减少显存和带宽需求。

### 2.1 数值格式详解

先理解各种数值格式的区别：

```
FP32 (32 bit):  1 符号 + 8 指数 + 23 尾数  → 范围大，精度高
FP16 (16 bit):  1 符号 + 5 指数 + 10 尾数  → 范围小，易溢出
BF16 (16 bit):  1 符号 + 8 指数 + 7 尾数   → 范围同 FP32，精度低于 FP16
FP8 E4M3:       1 符号 + 4 指数 + 3 尾数   → H100 引入，训练用
FP8 E5M2:       1 符号 + 5 指数 + 2 尾数   → H100 引入，推理用
INT8:           8 bit 整数 [-128, 127]
INT4:           4 bit 整数 [-8, 7]
```

**为什么训练用 BF16 而不是 FP16？**

BF16 的指数位和 FP32 一样（8 位），数值范围相同，不容易溢出。FP16 指数只有 5 位，范围小（max=65504），梯度更新时容易 overflow。LLaMA、GPT 等现代模型训练全部用 BF16。

### 2.2 量化基础

```
量化公式 (线性量化 / Uniform Quantization):

对称量化 (Symmetric):
  scale = max(|x|) / (2^(bits-1) - 1)
  x_quant = round(x / scale)
  x_dequant = x_quant × scale

非对称量化 (Asymmetric):
  scale = (max(x) - min(x)) / (2^bits - 1)
  zero_point = round(-min(x) / scale)
  x_quant = round(x / scale) + zero_point
  x_dequant = (x_quant - zero_point) × scale
```

**量化粒度**：

```
Per-tensor:  整个张量共用一个 scale           → 最粗，精度最差，速度最快
Per-channel: 每个输出通道一个 scale            → 中等，权重量化常用
Per-group:   每 g 个元素共用一个 scale (如 g=128) → 最细，精度最好
Per-token:   激活值按 token 维度各自一个 scale    → 激活量化常用
```

GPTQ 和 AWQ 都默认使用 **per-group (group_size=128)** 量化。

| 精度 | 每参数字节 | 7B 模型大小 | 相对精度损失 | 典型方法 |
|---|---|---|---|---|
| FP32 | 4 | 28 GB | 基准 | — |
| BF16 | 2 | 14 GB | 几乎无 | 训练标准 |
| INT8 | 1 | 7 GB | 很小 | SmoothQuant, LLM.int8() |
| INT4 | 0.5 | 3.5 GB | 可接受 | GPTQ, AWQ |
| NF4 | 0.5 | 3.5 GB | 比 INT4 更好 | QLoRA 专用 |

> **NF4 (NormalFloat4)**：QLoRA 提出的 4-bit 格式。假设权重近似正态分布，将量化桶设计为正态分布的分位数，比均匀 INT4 更匹配实际权重分布。

### 2.3 主流量化方法

#### GPTQ (Post-Training Quantization)

基于 OBQ（Optimal Brain Quantization），核心思想：**量化一个权重后，调整其余权重补偿误差**。

```
目标: min_Q ||WX - Q(W)X||²    ← 最小化量化前后输出的差异
      其中 X 是校准数据的激活值

等价于: min_Q ||W - Q(W)||²_H   ← Hessian 加权的权重误差
        其中 H = 2X·X^T (Hessian 矩阵，衡量每个权重的重要性)
```

逐列量化算法：

```
输入: 权重矩阵 W, Hessian 逆 H⁻¹, 目标位宽 b
输出: 量化后的权重 Q(W)

for i = 1 to d_out:           ← 逐列处理
    # 1. 量化第 i 列
    q_i = quantize(W[:, i], b)        ← 直接量化
    δ_i = W[:, i] - q_i               ← 量化误差

    # 2. 补偿剩余列（关键！）
    for j = i+1 to d_out:
        W[:, j] += δ_i × (H⁻¹[i, j] / H⁻¹[i, i])
        ↑ 把第 i 列的误差，按 Hessian 逆的比例分摊到后续列
        ↑ Hessian 逆告诉我们"调整哪些权重对输出影响最小"

    Q(W)[:, i] = q_i
```

**直觉理解**：量化 `w_i` 产生了误差 `δ`，Hessian 逆 `H⁻¹` 告诉我们在哪些方向调整影响最小。沿这些方向调整剩余权重，吸收 `δ` 的影响。

**GPTQ 的加速技巧**：
- 一次处理 128 列（block），block 内做密集矩阵运算（比逐列快 ~10x）
- Cholesky 分解加速 H⁻¹ 的计算
- 校准数据只需 128 条即可

#### AWQ (Activation-aware Weight Quantization)

**核心观察**：只有 ~1% 的权重通道对模型精度至关重要，这些通道对应的激活值特别大。

```
数学推导:

量化误差: Δ(w) = Round(w/s)·s - w     ← 最大约 s/2

对输出的影响: Δ(y) = Δ(w) · x          ← 误差 × 激活值

关键: 如果 x 很大 (重要通道)，即使 Δ(w) 很小，Δ(y) 也很大！

AWQ 的解决方案: 对重要通道的权重乘以 α > 1 再量化

  量化前: w → w·α  (放大)
  量化:   q = quantize(w·α)
  推理时: y = q · (x/α)  (激活侧除以 α 补偿)

  效果: 量化误差 Δ(w·α) ≈ s'/2 ，但对输出的影响 = s'/(2α) · x
        α 越大，重要通道的量化误差影响越小

最优 α 的选择:
  α* = (|x_mean| / max(|x_mean|))^p,  p ∈ [0, 1] 通过搜索确定
  即激活值越大的通道，α 越大（保护越强）
```

**AWQ vs GPTQ 对比**：

| | GPTQ | AWQ |
|---|---|---|
| 思路 | 量化后补偿（调权重） | 量化前保护（缩放通道）|
| 需要 Hessian | 是（计算量大） | 否 |
| 量化速度 | 几分钟~几小时 | 几分钟 |
| 精度 | 好 | 通常略好于 GPTQ |
| 推理速度 | 相同 | 相同 |

#### LLM.int8()

专门解决大模型中激活值 outlier 的 INT8 量化方案（bitsandbytes 库）：

```
观察: 6B+ 参数模型中，~0.1% 的激活维度值非常大 (outlier)
      这些 outlier 如果用 INT8 量化会严重损失精度

解决: 混合精度分解 (Mixed-precision Decomposition)
  1. 检测 outlier 维度 (|x| > threshold, 如 6.0)
  2. outlier 维度: 用 FP16 计算 (保精度)
  3. 其余维度: 用 INT8 计算 (省显存)
  4. 合并结果

  Y = X_outlier · W_outlier (FP16) + X_normal · W_normal (INT8)

outlier 通常只占 ~0.1% 维度 → 几乎不影响速度
```

#### GGUF / llama.cpp 量化

面向 CPU/边缘设备推理：

```
量化类型:
  Q4_0:   4-bit，每 32 个权重一个 scale              ← 最小，质量低
  Q4_K_M: 4-bit k-means 量化，不同层不同位宽        ← 推荐平衡
  Q5_K_M: 5-bit k-means                             ← 质量更好
  Q8_0:   8-bit                                      ← 质量接近原始

K-quant 的创新:
  对不同层用不同位宽:
  - 注意力层的 Q/K 用高位宽 (重要)
  - FFN 的中间层用低位宽 (冗余多)
  → 同样平均位宽下精度更好
```

### 2.4 Weight-Only vs Weight-Activation 量化

| | Weight-Only | Weight + Activation |
|---|---|---|
| 量化对象 | 只量化权重 | 权重和激活都量化 |
| 代表 | GPTQ, AWQ | SmoothQuant, W8A8 |
| 优势 | 简单，精度损失小 | 计算也加速（INT8 GEMM） |
| 难点 | 计算仍是 FP16 反量化 | 激活值有 outlier，量化难 |
| 适用场景 | 显存受限 | 吞吐量优先 |

**Weight-Only 量化的推理流程**：
```
存储: 权重以 INT4 存储 (省显存)
计算:
  1. 读取 INT4 权重
  2. 反量化 (Dequantize) 回 FP16
  3. 用 FP16 做矩阵乘

→ 省了显存和带宽，但计算本身没有加速
→ 对 Decode 阶段（访存瓶颈）非常有效：读取量减半 → 速度近乎翻倍
```

**SmoothQuant 详解**（W8A8 的关键）：

```
问题: 激活值某些通道有 outlier (值 >> 其他通道)
      ├── 权重: 分布平滑 ✓，容易量化
      └── 激活: 分布不平滑 ✗，直接量化损失大

SmoothQuant 的数学:
  Y = X · W = (X · diag(s)⁻¹) · (diag(s) · W)
      ↑ 不变     ↑ 激活除以 s      ↑ 权重乘以 s

  s 的选择 (per-channel):
  s_j = max(|X_j|)^α / max(|W_j|)^(1-α),  α ∈ [0.5, 0.75]
  → 把"不平滑"按比例从激活转移到权重
  → 两者都变得更容易量化

转移前:   激活范围 [0.1, 100]  权重范围 [-1, 1]    ← 激活难量化
转移后:   激活范围 [0.1, 5]    权重范围 [-20, 20]  ← 都比较均匀

转移后权重和激活都可以用 INT8 → W8A8 → 用 INT8 GEMM 加速计算
```

## 三、Speculative Decoding（投机解码）

### 3.1 核心思想

Decode 阶段的瓶颈是**访存**而非算力（GPU 算力没用满）。那么：

> 用一个小模型快速"猜"多个 token，再用大模型一次性**验证**这些 token。猜对了就赚到了，猜错了也不亏（回退到猜错的位置）。

```
传统 Decode (大模型 M):
  Step 1: M 生成 token₁        ← 1 次前向传播
  Step 2: M 生成 token₂        ← 1 次前向传播
  Step 3: M 生成 token₃        ← 1 次前向传播
  共 3 次大模型前向传播

Speculative Decode (小模型 m + 大模型 M):
  Step 1: m 连续猜 [t₁, t₂, t₃]    ← 3 次小模型前向（很快）
  Step 2: M 一次性验证 [t₁, t₂, t₃] ← 1 次大模型前向（并行验证）

  如果全猜对 → 等价于 1 次大模型调用生成了 3 个 token！
  如果 t₂ 猜错 → 接受 t₁，从 t₂ 位置重新开始
```

### 3.2 为什么验证可以并行

验证只是把猜测的 token 序列拼好，一次前向传播，检查大模型在每个位置的预测是否和猜测一致。这和 Prefill 一样是并行的。

```
验证过程 (猜了 γ=3 个 token):

输入:  [prompt..., t₁, t₂, t₃]    ← 把猜的 token 拼上
大模型一次前向传播, 输出每个位置的概率分布:
  P_M(·|prompt)         → 检查 t₁ 是否和 M 的采样一致
  P_M(·|prompt, t₁)     → 检查 t₂
  P_M(·|prompt, t₁, t₂) → 检查 t₃
  P_M(·|prompt, t₁, t₂, t₃) → 额外获得第 4 个 token（白赚的）

→ 一次大模型前向传播，最多验证 γ 个 + 白赚 1 个 = γ+1 个 token
```

### 3.3 拒绝采样保证分布一致性

**关键问题**：小模型的分布 $q(x)$ 和大模型 $p(x)$ 不同，怎么保证最终输出和大模型分布完全一致？

```
对每个位置 i，小模型猜了 token x_i ~ q(·):

接受概率: min(1, p(x_i) / q(x_i))

case 1: p(x_i) ≥ q(x_i)  → 必然接受（大模型也觉得这个 token 好）
case 2: p(x_i) < q(x_i)  → 以概率 p(x_i)/q(x_i) 接受

如果拒绝: 从修正分布 max(0, p(·) - q(·)) / Z 中重新采样
           → 补偿被小模型过度采样的部分

数学上可以证明: 这种方案的输出分布 = p(·)（和只用大模型完全一致）
```

### 3.4 接受率与加速比分析

```
接受率 α: 小模型单个 token 被大模型接受的平均概率
猜测长度 γ: 小模型每轮猜多少个 token

期望接受 token 数: E[accepted] = (1 - α^(γ+1)) / (1 - α)

加速比 ≈ E[accepted] / (小模型 γ 步 + 大模型 1 步 的时间比)

α = 0.7, γ = 5:  期望接受 ≈ 2.8 个 token/轮 → ~2.5x 加速
α = 0.9, γ = 5:  期望接受 ≈ 4.1 个 token/轮 → ~3.5x 加速
α = 0.5, γ = 5:  期望接受 ≈ 1.97 个/轮 → ~1.8x 加速

小模型越接近大模型 → α 越高 → 加速越大
但小模型太大 → 自身推理也慢 → 总加速可能反而降低
```

### 3.5 变体

| 方法 | 思路 | 是否需要额外模型 | 优势 |
|---|---|---|---|
| **标准投机** | 小模型 draft + 大模型验证 | 需要独立小模型 | 通用性强 |
| **Self-Speculative** | 大模型浅层 early exit 做 draft | 不需要 | 部署简单 |
| **Medusa** | 大模型加多个 draft head，每个预测不同位置 | 需要训 head | 无需小模型 |
| **Eagle** | 用大模型最后一层特征训练轻量 draft head | 需要训 head | 精度高 |
| **Lookahead** | 用 Jacobi 迭代并行生成多个 token | 不需要 | 无需训练 |

**Medusa 详细**：
```
标准 LM Head: 预测位置 t+1 的 token
Medusa Head 1: 预测位置 t+2 的 token  ← 额外加的 head
Medusa Head 2: 预测位置 t+3 的 token
...

每个 head 是一个小 MLP (hidden_dim → vocab_size)
训练时只训练这些 head (冻结主模型)

优势: 不需要独立的小模型，共享主模型的前向传播
劣势: 需要额外训练，且 head 预测越远精度越低
```

## 四、其他推理优化技术

### 4.1 Tensor Parallelism（张量并行）

将单个矩阵乘切分到多张 GPU 上，降低单卡显存和计算：

```
Attention:  W_q 按列切分 → GPU0 算 Q 的前半，GPU1 算后半
FFN:        W_up 按列切分，W_down 按行切分 → 最终 AllReduce 汇总
```

适用于 Prefill 阶段加速。

### 4.2 MoE 模型的推理部署

MoE 模型的推理与 Dense 模型有本质区别：**所有专家参数必须常驻显存**（因为路由是动态的），但每个 token 只激活少数专家。这带来了独特的部署挑战。

#### 核心问题：显存大，但计算稀疏

```
Dense 7B:   加载 7B 参数，每个 token 用 7B 参数 → 显存和计算匹配
MoE 47B:    加载 47B 参数，每个 token 只用 13B → 显存远大于计算需求
```

单卡放不下全部专家时，必须把专家分散到多块 GPU 上——这就是**专家并行 (Expert Parallelism, EP)**。

#### 专家并行的通信流程

以 4 块 GPU 部署 8 个路由专家 + 1 个共享专家、TopK=2 为例：

```
GPU 0: 共享专家(副本) + E0, E1
GPU 1: 共享专家(副本) + E2, E3
GPU 2: 共享专家(副本) + E4, E5
GPU 3: 共享专家(副本) + E6, E7
```

一个 batch 的 8 个 token 分布在 4 块 GPU 上，推理过程如下：

**① 共享专家：本地计算，无通信**

```
GPU 0 上的 t1, t2 → 本地共享专家副本处理 → y_shared
GPU 1 上的 t3, t4 → 本地共享专家副本处理 → y_shared
GPU 2 上的 t5, t6 → 本地共享专家副本处理 → y_shared
GPU 3 上的 t7, t8 → 本地共享专家副本处理 → y_shared
```

共享专家每块 GPU 都有一份副本，和 Attention 层一样用**数据并行**处理，不需要跨 GPU 通信。

**② 路由决策 + All-to-All 通信**

路由器在每块 GPU 上本地计算 token 的路由结果，然后 token 需要被**发送到对应专家所在的 GPU**：

```
GPU 0 上的路由结果:
  t1 → E0(本地), E5(GPU 2)     ← t1 需要发送到 GPU 2
  t2 → E3(GPU 1), E7(GPU 3)   ← t2 需要发送到 GPU 1 和 GPU 3

GPU 1 上的路由结果:
  t3 → E0(GPU 0), E1(GPU 0)   ← t3 需要发送到 GPU 0
  t4 → E4(GPU 2), E6(GPU 3)   ← t4 需要发送到 GPU 2 和 GPU 3
  ...
```

所有 GPU 同时互相发送/接收 token，这就是 **All-to-All 通信**——通信模式完全由路由结果动态决定，无法提前优化。

**③ 专家计算**

各 GPU 上的专家处理收到的 token。由于路由的动态性，各专家的负载可能不均：

```
E0: ████████ 4 个 token
E1: █████ 3 个 token
E2:  0 个 token          ← 空闲
E5: ████████████ 6 个 token  ← 最慢的专家决定整体速度
```

所有 GPU 必须同步等最慢的专家完成，形成**木桶效应**。

**④ All-to-All 通信（返回结果）**

专家计算完成后，结果通过第二次 All-to-All 送回原来的 GPU，与共享专家输出相加：

```
最终输出 = y_shared（本地） + y_routed（通信返回）
```

#### 三类组件的部署特性对比

| | 共享专家 | 路由专家 | 路由器 |
|---|---|---|---|
| **部署方式** | 每块 GPU 一份副本 | 分散在不同 GPU | 每块 GPU 一份副本 |
| **通信需求** | 无（本地计算） | 两次 All-to-All | 无 |
| **负载均衡** | 不需要（天然均匀） | 核心难题 | 不涉及 |
| **类比** | 像 Attention 层 | 像分布式微服务 | 像 API Gateway |

#### MoE 推理优化策略

- **Expert Offloading**：将不常用的专家卸载到 CPU/SSD，需要时加载回 GPU（增加延迟，但降低显存需求）
- **Expert Caching**：根据历史统计预测即将使用的专家，提前加载到 GPU
- **量化**：对专家参数做 INT4/INT8 量化，减少显存占用和通信数据量（DeepSeek-V3 使用 FP8）
- **通信与计算重叠**：在等待 All-to-All 通信时，提前计算共享专家或 Attention 层

### 4.3 多 LoRA 服务

vLLM 等框架支持同一个 base model 挂载多个 LoRA adapter，根据请求路由到不同 LoRA，共享 base model 的 KV Cache 和权重。

**面试快速回答模板**：

> **Q: FlashAttention 为什么快？**
> A: 避免把 N×N 的注意力矩阵写入 HBM。通过 tiling 在 SRAM 里分块计算，配合 online softmax 流式更新，HBM 访问量从 O(N²) 降到 O(N²d/M)。
>
> **Q: 量化会损失精度吗？怎么减少？**
> A: INT8 损失很小，INT4 有可感知的损失。GPTQ 通过量化后补偿剩余权重、AWQ 通过保护重要通道来减少损失。
