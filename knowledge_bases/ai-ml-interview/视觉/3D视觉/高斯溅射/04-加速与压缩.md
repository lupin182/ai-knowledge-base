# 04 · 加速与压缩

3DGS 出名的两个数字：**渲染快**（>100 FPS）但 **存储大**（百万级高斯，几百 MB）。这一篇讲 2024 年里把 GS 从"研究 demo"推到"能上线"的几条工程线：抗混叠、层次化锚点、压缩编码、移动端部署。

> **核心论文**:
> - Mip-Splatting (Yu et al., CVPR 2024)
> - Scaffold-GS (Lu et al., CVPR 2024)
> - Octree-GS (Ren et al., 2024)
> - Compact3D / LightGaussian (2024)
> - HAC (Chen et al., ECCV 2024)

---

## 一、3DGS 的工程痛点

| 痛点 | 表现 | 解决方向 |
|------|-----|---------|
| 抗混叠差 | zoom in/out 时纹理闪烁、远处糊近处锯齿 | 多尺度滤波（Mip-Splatting） |
| 内存大 | 1080p 中等场景 ~500 MB | 锚点结构 + 量化（Scaffold-GS / HAC） |
| 高斯数爆炸 | adaptive density control 让数量失控 | 重要性剪枝（LightGaussian） |
| 远场渲染慢 | 远处视角下高斯密度过高 | LOD / 层次化（Octree-GS） |
| 移动端跑不动 | 缺 CUDA、内存只有几 GB | WebGL 渲染 + 极致压缩 |

## 二、Mip-Splatting：抗混叠

### 2.1 问题

3DGS 训练时只见过某个分辨率/某个相机距离的视角。换分辨率/换距离，原本"刚好不混叠"的高斯就会出问题：

- **远场放大**：高斯太小，被映射到一个像素以下，造成欠采样、闪烁
- **近场缩小**：高斯太大，被覆盖到大量像素，造成模糊或块状 artifact

类比 NeRF 的 mip-NeRF——同一个问题在 GS 上更明显。

### 2.2 两个滤波

Mip-Splatting 在 3DGS 训练 + 渲染中加了两个滤波：

- **3D Smoothing Filter**：对每个高斯做下界约束——它的最小尺寸不能小于 $r_{\min}/d_i$，其中 $d_i$ 是高斯到训练相机的最近距离。这阻止训练把高斯"挤到无限小"以拟合某个特定像素。
- **2D Mip Filter**：渲染时根据当前视角的"屏幕空间像素覆盖比"动态加一个 2D Gaussian 滤波 $\sigma_{\text{mip}}$ 给屏幕空间协方差：
  $$\Sigma'_{2D} = \Sigma_{2D} + \sigma_{\text{mip}}^2 \cdot I$$

数学上等价于在屏幕空间做一个 prefilter，避免欠采样混叠。

### 2.3 效果

- 远场放大不再闪烁
- 训练分辨率与渲染分辨率不同时质量稳定
- 核心代码改动很小（rasterizer 里加几行），向后兼容
- 几乎所有 2024 后续 GS 工作都默认开启 Mip-Splatting

## 三、Scaffold-GS：锚点 + view-dependent MLP

### 3.1 思路

3DGS 里每个高斯都是独立的 free parameter，参数量 = 高斯数 × 59。Scaffold-GS 改成：

- 场景中放 **anchor point**（锚点），每个 anchor 关联**一组**（如 $K=10$ 个）高斯
- 锚点存：位置、特征向量、上下文（局部偏移、缩放、旋转）
- 每个高斯的属性（颜色、不透明度）由一个**小 MLP** 从锚点特征 + 视角方向预测：
  $$\alpha_k, c_k = \text{MLP}(\mathbf{f}_{\text{anchor}}, \mathbf{d}_{\text{view}}, \boldsymbol{\Delta}_k)$$

### 3.2 优势

- **参数大幅减少**：原来每个高斯都存自己的 59 维参数，现在一个锚点服务 K 个高斯，参数量 / K
- **view-dependent 更优雅**：不再用球谐函数（48 系数），而是 view 进 MLP 直接预测颜色，能表达更复杂的高频反射
- **结构化**：锚点天然支持空间索引、剔除、LOD

### 3.3 训练

锚点初始化：从 SfM 点云。训练时分两步迭代：

1. 锚点位置 / 特征 / 上下文用 Adam 优化
2. 高斯属性由 MLP 即时计算（无需存储）

成长策略：根据高斯被使用频率，**生长**（在频繁覆盖区添加锚点）或**剪枝**（移除冗余锚点）。

### 3.4 与原版 3DGS 对比

| 维度 | 3DGS | Scaffold-GS |
|------|------|-------------|
| 参数量（同等质量） | 100% | ~30-40% |
| 反射 / 高光质量 | 一般 | 更好（MLP 表达 view-dep） |
| 训练时间 | 30 min | 40 min |
| 渲染速度 | 100+ FPS | 80+ FPS（MLP 推理略增） |

Scaffold-GS 是 2024 后半年压缩流派的主流 backbone。

## 四、Octree-GS：层次化 LOD

### 4.1 思路

Scaffold-GS 的锚点是平面结构，远处仍然查所有锚点。Octree-GS 把锚点放进**八叉树**：

- 不同深度的节点对应不同空间分辨率
- 渲染时根据当前视角的距离 / 屏幕空间像素覆盖，选择合适深度的锚点

类比游戏引擎里的 LOD，但是数据驱动——每个节点的特征通过训练学到。

### 4.2 效果

- 远场渲染加速：不用绘制远处的小高斯
- 大场景内存友好：只把视锥内 + 当前 LOD 的锚点装进显存
- 自动避免远场混叠（和 Mip-Splatting 互补）

### 4.3 适合场景

- **城市级场景**（无人机航拍、Google Earth 风格）
- **大型室内场景**（商场、博物馆）
- 普通的物体级场景没必要做层次化

## 五、压缩流派

锚点结构降低了参数量，但每个 float 还是 32-bit。压缩流派进一步用**量化 + 编码**降存储。

### 5.1 LightGaussian

- **重要性评分剪枝**：根据每个高斯对最终渲染的贡献（光线被吸收的总量）评分，剪掉低分高斯
- **球谐蒸馏**：把高阶 SH 系数（48 维）蒸馏成低阶 + 残差 MLP
- **向量量化**：剩余 SH 系数用 k-means 量化到 codebook
- **结果**：~15× 压缩，质量损失 < 1 dB

### 5.2 Compact3D

- 类似 LightGaussian，但向量量化更激进
- 加 **稀疏化**：让大部分高斯的高阶 SH 接近 0，再用稀疏编码存储
- ~10-20× 压缩

### 5.3 HAC（Hash-based Anchor Compression）

- 基于 Scaffold-GS 的锚点结构
- 锚点位置走**哈希网格**编码（类似 Instant-NGP）
- 锚点特征做**上下文熵编码**：相邻锚点特征相似 → 用前一个锚点预测当前锚点的概率分布 → 算术编码
- ~75× 压缩，质量基本无损（< 0.3 dB 损失）
- 是 2024 年压缩 SOTA

### 5.4 其他

- **EAGLES**：Efficient Anchor + Lossless Encoding，无损版本
- **Reduced3DGS**：训练时直接学少数高斯
- **Self-Organizing Gaussians**：把高斯属性映射到 2D 网格，按图像压缩

## 六、移动端 / Web

### 6.1 WebGL 渲染

- **antimatter15/splat**：第一个 WebGL 3DGS 渲染器，仅前端，完全不要 CUDA
- **gsplat.js**：JavaScript 库，更易集成
- **SuperSplat**：浏览器内编辑器，支持高斯增删改

工程要点：
- WebGL 没有 radix sort，要用 GPU compute / WebGPU 实现 tile-based 排序
- 浮点精度问题：WebGL 默认 float16，远处高斯精度不够
- 内存：浏览器 tab 通常限制 1-2 GB，必须配合压缩（HAC + 量化）

### 6.2 移动端

- **Splat-on-Mobile**：iOS / Android 原生渲染器，用 Metal / Vulkan
- **GS Player**：苹果 Vision Pro 上的高斯播放器
- 关键：模型必须 < 50 MB（手机内存限制），需要 HAC 级别压缩

## 七、组合策略对比

如果你做新项目，怎么组合这些技术？

| 场景 | 推荐组合 | 理由 |
|------|---------|------|
| 单物体高质量渲染 | 3DGS + Mip-Splatting | 简单，质量优先 |
| 室内大场景 | Scaffold-GS + Mip-Splatting | 锚点降参，反射更好 |
| 城市级 / 航拍 | Octree-GS + Scaffold-GS | LOD 必要 |
| 上线 / Web 部署 | Scaffold-GS + HAC + WebGL | 极致压缩 + 浏览器渲染 |
| 移动端 | HAC + Reduced3DGS + Metal/Vulkan | 内存与算力双重限制 |

## 八、面试高频问题

### Q1: Mip-Splatting 解决的是什么问题？

**答**：抗混叠。3DGS 训练时只见过特定分辨率/距离的视角，换分辨率或换距离会因欠采样/过采样产生闪烁与模糊。Mip-Splatting 加了两个滤波：3D smoothing filter 给高斯尺寸下界（不能小于一个像素覆盖），2D mip filter 在屏幕空间给协方差加 prefilter $\sigma^2 I$，等价于训练时就把所有可能的渲染分辨率纳入考虑。

### Q2: Scaffold-GS 为什么能减参数？

**答**：原版每个高斯独立存 59 维参数；Scaffold-GS 把若干高斯绑到一个 anchor，anchor 存特征向量，高斯属性由 MLP 即时算。参数量 ≈ anchor 数 × (位置 + 特征 + 几个偏移) + MLP 权重，比"高斯数 × 59"小很多。代价是渲染时多一次 MLP 推理。

### Q3: HAC 压缩比 LightGaussian 高这么多，靠的是什么？

**答**：HAC 的关键在**上下文熵编码**——相邻锚点特征高度相关，用前一个锚点的特征预测当前锚点的概率分布，再算术编码。这个想法来自图像/视频压缩（HEVC/H.266），把"信号本身"压缩 → "残差/预测误差"压缩，压缩比上一个量级。LightGaussian 只做向量量化，没有上下文模型。

### Q4: 我要把 3DGS 部署到 web 上让用户在浏览器里看，最该解决什么？

**答**：(1) **压缩**：HAC + 量化把模型从 500 MB 压到 < 50 MB（移动端 < 20 MB）；(2) **WebGL/WebGPU 排序**：原版 CUDA radix sort 不能用，要写 GPU compute 版的 tile-based bitonic sort；(3) **浮点精度**：WebGL float16 在远场高斯上不够，必要时退回 CPU 排序。

### Q5: Octree-GS 和 Scaffold-GS 是替代关系还是互补？

**答**：互补。Scaffold-GS 解决"参数量大"——把高斯结构化到锚点；Octree-GS 解决"远场算力浪费 + 大场景"——把锚点放进八叉树做 LOD。两者通常一起用：Scaffold 锚点放进 Octree 节点。
